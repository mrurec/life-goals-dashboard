Введение: Netflix DGS для Life Goals Dashboard

Netflix DGS 11.1.0 (Domain Graph Service) — это Java фреймворк для построения GraphQL API, который тесно интегрируется с Spring Boot. В Life Goals Dashboard мы используем DGS для реализации всех GraphQL resolver'ов, которые служат "мостом" между клиентом (React) и backend сервисами (Kotlin).

За 3 дня вы освоите:

  • Schema-first подход: писание `.graphqls` файлов, генерация типов через `./gradlew generateJava`
  • Resolver'ы: `@DgsQuery`, `@DgsMutation`, `@DgsData` для распределения логики
  • DataLoaders для предотвращения N+1 проблем
  • Subscriptions и WebSocket для real-time уведомлений
  • Обработка ошибок через `Payload` типы с массивом `[UserError!]!`
Совет: DGS упрощает работу с GraphQL в Spring Boot благодаря аннотациям. Код становится чище и более декларативным.
DGS 11 + Spring Boot 4: DGS 11.1.0 встроен на Spring Boot 4.0.5 — аннотации `@DgsComponent`, `@DgsQuery`, `@DgsMutation` остаются без изменений; новое: улучшенная поддержка JSpecify nullability и Virtual Threads-aware dispatchers.

Schema-first подход

Основной принцип: .graphqls → Kotlin типы

В Life Goals Dashboard мы следуем schema-first подходу:

  1. Пишете GraphQL схему в `.graphqls` файл (в `src/main/resources/schema/`)
  2. Запускаете `./gradlew generateJava` — DGS генерирует Kotlin типы
  3. Реализуете resolver'ы, которые возвращают эти типы
  4. Сохраняете сгенерированные файлы в `.gitignore` в папке `__generated__/`

Преимущества: Schema это истина. Типы всегда синхронизированы. Клиент и сервер говорят на одном языке.

Пример: Fitness Widget Schema

Создайте файл `src/main/resources/schema/fitness.graphqls`:

Генерация типов

После написания схемы запустите:


./gradlew generateJava

DGS генерирует файлы в `build/generated/sources/dgs/` — например, `Workout.kt`, `ExerciseSet.kt` и т.д. Эти файлы содержат data-классы, которые используются в resolver'ах.

Важно: Никогда не редактируйте сгенерированные файлы вручную. Измените схему и перегенерируйте.

Resolvers: @DgsQuery, @DgsMutation, @DgsData

Основная концепция

Resolver — это функция, которая "решает" (resolve) поле в GraphQL типе. Она принимает входные аргументы и возвращает значение.

@DgsQuery и @DgsMutation

Это resolvers для корневых типов Query и Mutation. Каждый resolver — это метод в `@DgsComponent` классе. 


import com.netflix.graphql.dgs.DgsComponent
import com.netflix.graphql.dgs.DgsQuery
import com.netflix.graphql.dgs.DgsMutation
import com.netflix.graphql.dgs.DgsDataFetchingEnvironment
import com.mrurec.lifegoals.fitness.model.Workout
import com.mrurec.lifegoals.fitness.service.WorkoutService
import com.mrurec.lifegoals.common.auth.ViewerContext
import org.springframework.beans.factory.annotation.Autowired

@DgsComponent
class FitnessResolver(
    @Autowired val workoutService: WorkoutService
) {

    @DgsQuery
    fun myWorkouts(
        dfe: DgsDataFetchingEnvironment,
        first: Int,
        after: String?
    ): WorkoutConnection {
        val viewer = dfe.getLocalContext()
        return workoutService.getWorkoutsForUser(
            userId = viewer.userId,
            first = first,
            after = after
        )
    }

    @DgsQuery
    fun workoutDetail(
        dfe: DgsDataFetchingEnvironment,
        id: String
    ): Workout? {
        val viewer = dfe.getLocalContext()
        return workoutService.getWorkout(viewer.userId, id)
    }

    @DgsMutation
    fun logWorkout(
        dfe: DgsDataFetchingEnvironment,
        input: LogWorkoutInput
    ): LogWorkoutPayload {
        val viewer = dfe.getLocalContext()

        // Валидация
        val errors = mutableListOf()
        if (input.durationMinutes <= 0) {
            errors.add(UserError(
                field = "durationMinutes",
                message = "Duration must be positive",
                code = ErrorCode.VALIDATION_ERROR
            ))
        }

        if (errors.isNotEmpty()) {
            return LogWorkoutPayload(workout = null, errors = errors)
        }

        try {
            val workout = workoutService.createWorkout(
                userId = viewer.userId,
                exerciseName = input.exerciseName,
                durationMinutes = input.durationMinutes,
                caloriesBurned = input.caloriesBurned
            )
            return LogWorkoutPayload(workout = workout, errors = emptyList())
        } catch (e: Exception) {
            return LogWorkoutPayload(
                workout = null,
                errors = listOf(UserError(
                    field = null,
                    message = "Failed to log workout: ${e.message}",
                    code = ErrorCode.INTERNAL_ERROR
                ))
            )
        }
    }
}

@DgsData для child resolvers

Когда нужно resolve поля, которое не является Query или Mutation, используйте `@DgsData`. Это для вложенных типов. 


@DgsComponent
class WorkoutResolver(
    @Autowired val exerciseService: ExerciseService
) {

    // Resolve поле "sets" у типа Workout
    @DgsData(parentType = "Workout", field = "sets")
    fun exerciseSets(
        dfe: DgsDataFetchingEnvironment
    ): CompletableFuture> {
        val workout = dfe.getSource()

        // Используйте DataLoader для batching (см. секция ниже)
        val loader = dfe.getDataLoader>("exerciseSetsLoader")
        return loader.load(workout.id)
    }
}

DgsDataFetchingEnvironment

Это объект, который содержит всю информацию о текущем запросе:

  • `getLocalContext()` — получить пользовательский контекст (ViewerContext)
  • `getSource()` — получить родительский объект (для child resolvers)
  • `getArgument("name")` — получить аргумент запроса
  • `getDataLoader("name")` — получить DataLoader для batching

DataLoaders — N+1 Prevention

Проблема N+1

Представьте запрос: "Дайте мне 20 тренировок, и для каждой все упражнения (sets)". Наивный resolver может сделать 1 запрос для тренировок + 20 запросов для упражнений = 21 SQL запрос (N+1 проблема).

DataLoader батчит (объединяет) запросы: 1 запрос для тренировок + 1 запрос для всех упражнений = 2 запроса.

Реализация DataLoader

Создайте `ExerciseSetsDataLoader.kt`:


import com.netflix.graphql.dgs.DgsComponent
import com.netflix.graphql.dgs.DgsDataLoader
import com.mrurec.lifegoals.fitness.model.ExerciseSet
import com.mrurec.lifegoals.fitness.repository.ExerciseSetRepository
import org.dataloader.MappedBatchLoader
import java.util.*
import java.util.concurrent.CompletableFuture

@DgsComponent
class ExerciseSetsDataLoader(
    val repository: ExerciseSetRepository
) {

    @DgsDataLoader(name = "exerciseSetsLoader")
    fun batchLoader(): MappedBatchLoader> {
        return MappedBatchLoader { workoutIds, _ ->
            // Один SQL запрос для всех workout ID'ов
            val sets = repository.findByWorkoutIdIn(workoutIds)

            // Результат: Map>
            val result = mutableMapOf>()
            workoutIds.forEach { workoutId ->
                result[workoutId] = sets.filter { it.workoutId == workoutId }
            }

            CompletableFuture.completedFuture(result)
        }
    }
}

Использование DataLoader в resolver'е

Регистрация DataLoader'ов

В Spring Boot приложении DataLoader'ы регистрируются автоматически через `@DgsDataLoader` аннотацию. DGS сканирует класс пути и находит все такие компоненты.

Совет: DataLoader'ы существуют только на время одного GraphQL запроса. Это гарантирует, что батчинг работает корректно, но разные запросы используют разные экземпляры.

Subscriptions: WebSocket и Real-time

Что такое Subscription?

В отличие от Query (единовременный запрос) и Mutation (изменение + ответ), Subscription — это долгоживущая связь, где сервер может отправлять данные клиенту в реальном времени.

В Life Goals Dashboard используем Subscription'ы для:

  • Уведомлений о достижениях (achievements)
  • Обновлений о новых записях (workouts, meals, transactions)
  • Статус изменений виджетов

Schema для Subscription


type Subscription {
    # Подписка на уведомления пользователя
    notificationReceived: Notification!

    # Подписка на обновления виджета
    widgetDataUpdated(widgetId: ID!): WidgetData!
}

type Notification {
    id: ID!
    type: NotificationType!
    title: String!
    body: String!
    createdAt: DateTime!
}

enum NotificationType {
    ACHIEVEMENT
    REMINDER
    WARNING
    MILESTONE
}

Реализация Subscription resolver'а


import com.netflix.graphql.dgs.DgsSubscription
import com.netflix.graphql.dgs.DgsComponent
import com.mrurec.lifegoals.notification.model.Notification
import com.mrurec.lifegoals.notification.publisher.NotificationPublisher
import org.reactivestreams.Publisher
import reactor.core.publisher.Flux

@DgsComponent
class NotificationSubscription(
    val notificationPublisher: NotificationPublisher
) {

    @DgsSubscription
    fun notificationReceived(): Publisher {
        // Возвращает Flux — stream уведомлений
        return notificationPublisher.getNotificationStream()
    }
}

NotificationPublisher (Redis pub/sub backed)


import org.springframework.data.redis.core.StringRedisTemplate
import org.springframework.stereotype.Service
import reactor.core.publisher.Flux
import reactor.core.publisher.Sinks
import com.mrurec.lifegoals.common.auth.ViewerContext
import com.mrurec.lifegoals.notification.model.Notification

@Service
class NotificationPublisher(
    val redisTemplate: StringRedisTemplate
) {

    private val sinks = mutableMapOf>()

    fun publishNotification(userId: String, notification: Notification) {
        // Публикуем в Redis канал
        val channel = "lifegoals:notifications:$userId"
        redisTemplate.convertAndSend(channel, notification)

        // Также отправляем в Flux для WebSocket подключений
        sinks[userId]?.tryEmitNext(notification)
    }

    fun getNotificationStream(): Flux {
        // Создаём Flux, который отправляет уведомления
        val sink = Sinks.many().multicast().onBackpressureBuffer()
        // Код упрощён; в реальности нужна подписка на Redis канал
        return sink.asFlux()
    }
}

Best Practices: Обработка ошибок и валидация

Принцип: Ошибки в Payload, не в GraphQL errors

В Meta системах (и в этом проекте) бизнес-логические ошибки возвращаются в `Payload` объекте, а не выбрасываются как GraphQL ошибки. Это позволяет фронтенду обрабатывать ошибки элегантно. 


# ПРАВИЛЬНО
type LogWorkoutPayload {
    workout: Workout
    errors: [UserError!]!
}

type UserError {
    field: String
    message: String!
    code: ErrorCode!
}

enum ErrorCode {
    VALIDATION_ERROR
    NOT_FOUND
    UNAUTHORIZED
    INTERNAL_ERROR
}

Фронтенд проверяет `payload.errors` и отображает их: 


const [logWorkout, isLoading] = useMutation(LogWorkoutMutation);

const handleLogWorkout = async (input) => {
    const response = await logWorkout({ variables: { input } });

    if (response.data.logWorkout.errors.length > 0) {
        // Показать ошибки пользователю
        response.data.logWorkout.errors.forEach(err => {
            console.error(err.message);
        });
    } else {
        // Успешно добавлена тренировка
        console.log("Workout logged:", response.data.logWorkout.workout);
    }
};

Валидация на сервере


@DgsMutation
fun createRecipe(
    dfe: DgsDataFetchingEnvironment,
    input: CreateRecipeInput
): CreateRecipePayload {
    val viewer = dfe.getLocalContext()
    val errors = mutableListOf()

    // Валидация 1: name не пустое
    if (input.name.trim().isEmpty()) {
        errors.add(UserError(
            field = "name",
            message = "Recipe name cannot be empty",
            code = ErrorCode.VALIDATION_ERROR
        ))
    }

    // Валидация 2: минимум 1 ингредиент
    if (input.ingredients.isEmpty()) {
        errors.add(UserError(
            field = "ingredients",
            message = "Recipe must have at least one ingredient",
            code = ErrorCode.VALIDATION_ERROR
        ))
    }

    // Валидация 3: проверка существования ингредиентов
    val invalidIngredients = input.ingredients.filter {
        !recipeService.productExists(viewer.userId, it.productId)
    }
    if (invalidIngredients.isNotEmpty()) {
        errors.add(UserError(
            field = "ingredients",
            message = "Some products not found",
            code = ErrorCode.NOT_FOUND
        ))
    }

    if (errors.isNotEmpty()) {
        return CreateRecipePayload(recipe = null, errors = errors)
    }

    // Создание рецепта если валидация прошла
    val recipe = recipeService.createRecipe(viewer.userId, input)
    return CreateRecipePayload(recipe = recipe, errors = emptyList())
}

package com.mrurec.lifegoals.common.graphql

import com.netflix.graphql.dgs.DgsQueryExecutor
import org.junit.jupiter.api.Test
import org.junit.jupiter.api.Assertions.*
import org.springframework.beans.factory.annotation.Autowired
import org.springframework.boot.test.context.SpringBootTest

@SpringBootTest
class GraphQLExceptionHandlerTest {

    @Autowired
    lateinit var dgsQueryExecutor: DgsQueryExecutor

    @Test
    fun `unauthenticated request should return UNAUTHENTICATED error code`() {
        // Execute a query without auth token
        val result = dgsQueryExecutor.execute("{ myDashboard { widgets { id } } }")

        assertTrue(result.errors.isNotEmpty())
        val errorCode = result.errors[0].extensions?.get("errorType")
        assertEquals("UNAUTHENTICATED", errorCode)

    @Test
    fun `should not expose stack traces in error response`() {
        val result = dgsQueryExecutor.execute("{ myDashboard { widgets { id } } }")

        for (error in result.errors) {
            assertNull(error.extensions?.get("stackTrace"),
                "Stack traces should not be exposed to clients")
        }
    }
}

Introduction: Netflix DGS for Life Goals Dashboard

Netflix DGS 11.1.0 (Domain Graph Service) is a Java framework for building GraphQL APIs that integrates tightly with Spring Boot. In Life Goals Dashboard, we use DGS to implement all GraphQL resolvers, which serve as a "bridge" between the client (React) and backend services (Kotlin).

Over 3 days you will master:

  • Schema-first approach: writing `.graphqls` files, generating types via `./gradlew generateJava`
  • Resolvers: `@DgsQuery`, `@DgsMutation`, `@DgsData` for logic distribution
  • DataLoaders to prevent N+1 problems
  • Subscriptions and WebSocket for real-time notifications
  • Error handling via `Payload` types with `[UserError!]!` array
Tip: DGS simplifies GraphQL work in Spring Boot through annotations. Code becomes cleaner and more declarative.
DGS 11 + Spring Boot 4: DGS 11.1.0 is built on Spring Boot 4.0.5 — `@DgsComponent`, `@DgsQuery`, `@DgsMutation` remain unchanged; new features include better JSpecify nullability support and Virtual Threads-aware dispatchers.

Schema-first Approach

Core principle: .graphqls → Kotlin types

In Life Goals Dashboard we follow the schema-first approach:

  1. Write GraphQL schema in `.graphqls` file (in `src/main/resources/schema/`)
  2. Run `./gradlew generateJava` — DGS generates Kotlin types
  3. Implement resolvers that return these types
  4. Keep generated files in `.gitignore` in `__generated__/` folder

Benefits: Schema is the source of truth. Types are always synchronized. Client and server speak the same language.

Type Generation

After writing the schema, run:


./gradlew generateJava

DGS generates files in `build/generated/sources/dgs/` — for example, `Workout.kt`, `ExerciseSet.kt`, etc. These files contain data classes used in resolvers.

Important: Never manually edit generated files. Change the schema and regenerate.

Resolvers: @DgsQuery, @DgsMutation, @DgsData

Core concept

A resolver is a function that "resolves" (fetches) a field in a GraphQL type. It takes input arguments and returns a value.

@DgsQuery and @DgsMutation

These are resolvers for the root Query and Mutation types. Each resolver is a method in a `@DgsComponent` class. 


import com.netflix.graphql.dgs.DgsComponent
import com.netflix.graphql.dgs.DgsQuery
import com.netflix.graphql.dgs.DgsMutation
import com.netflix.graphql.dgs.DgsDataFetchingEnvironment
import com.mrurec.lifegoals.fitness.model.Workout
import com.mrurec.lifegoals.fitness.service.WorkoutService
import com.mrurec.lifegoals.common.auth.ViewerContext
import org.springframework.beans.factory.annotation.Autowired

@DgsComponent
class FitnessResolver(
    @Autowired val workoutService: WorkoutService
) {

    @DgsQuery
    fun myWorkouts(
        dfe: DgsDataFetchingEnvironment,
        first: Int,
        after: String?
    ): WorkoutConnection {
        val viewer = dfe.getLocalContext()
        return workoutService.getWorkoutsForUser(
            userId = viewer.userId,
            first = first,
            after = after
        )
    }

    @DgsQuery
    fun workoutDetail(
        dfe: DgsDataFetchingEnvironment,
        id: String
    ): Workout? {
        val viewer = dfe.getLocalContext()
        return workoutService.getWorkout(viewer.userId, id)
    }

    @DgsMutation
    fun logWorkout(
        dfe: DgsDataFetchingEnvironment,
        input: LogWorkoutInput
    ): LogWorkoutPayload {
        val viewer = dfe.getLocalContext()

        // Validation
        val errors = mutableListOf()
        if (input.durationMinutes <= 0) {
            errors.add(UserError(
                field = "durationMinutes",
                message = "Duration must be positive",
                code = ErrorCode.VALIDATION_ERROR
            ))
        }

        if (errors.isNotEmpty()) {
            return LogWorkoutPayload(workout = null, errors = errors)
        }

        try {
            val workout = workoutService.createWorkout(
                userId = viewer.userId,
                exerciseName = input.exerciseName,
                durationMinutes = input.durationMinutes,
                caloriesBurned = input.caloriesBurned
            )
            return LogWorkoutPayload(workout = workout, errors = emptyList())
        } catch (e: Exception) {
            return LogWorkoutPayload(
                workout = null,
                errors = listOf(UserError(
                    field = null,
                    message = "Failed to log workout: ${e.message}",
                    code = ErrorCode.INTERNAL_ERROR
                ))
            )
        }
    }
}

@DgsData for child resolvers

When you need to resolve a field that is not Query or Mutation, use `@DgsData`. This is for nested types. 


@DgsComponent
class WorkoutResolver(
    @Autowired val exerciseService: ExerciseService
) {

    // Resolve "sets" field of Workout type
    @DgsData(parentType = "Workout", field = "sets")
    fun exerciseSets(
        dfe: DgsDataFetchingEnvironment
    ): CompletableFuture> {
        val workout = dfe.getSource()

        // Use DataLoader for batching (see section below)
        val loader = dfe.getDataLoader>("exerciseSetsLoader")
        return loader.load(workout.id)
    }
}

DgsDataFetchingEnvironment

This object contains all information about the current request:

  • `getLocalContext()` — get user context (ViewerContext)
  • `getSource()` — get parent object (for child resolvers)
  • `getArgument("name")` — get request argument
  • `getDataLoader("name")` — get DataLoader for batching

DataLoaders — N+1 Prevention

The N+1 Problem

Imagine a query: "Give me 20 workouts, and for each one all exercises (sets)". A naive resolver might make 1 query for workouts + 20 queries for exercises = 21 SQL queries (N+1 problem).

DataLoader batches (combines) queries: 1 query for workouts + 1 query for all exercises = 2 queries.

DataLoader Implementation

Create `ExerciseSetsDataLoader.kt`:


import com.netflix.graphql.dgs.DgsComponent
import com.netflix.graphql.dgs.DgsDataLoader
import com.mrurec.lifegoals.fitness.model.ExerciseSet
import com.mrurec.lifegoals.fitness.repository.ExerciseSetRepository
import org.dataloader.MappedBatchLoader
import java.util.*
import java.util.concurrent.CompletableFuture

@DgsComponent
class ExerciseSetsDataLoader(
    val repository: ExerciseSetRepository
) {

    @DgsDataLoader(name = "exerciseSetsLoader")
    fun batchLoader(): MappedBatchLoader> {
        return MappedBatchLoader { workoutIds, _ ->
            // Single SQL query for all workout IDs
            val sets = repository.findByWorkoutIdIn(workoutIds)

            // Result: Map>
            val result = mutableMapOf>()
            workoutIds.forEach { workoutId ->
                result[workoutId] = sets.filter { it.workoutId == workoutId }
            }

            CompletableFuture.completedFuture(result)
        }
    }
}

Using DataLoader in resolver


@DgsComponent
class WorkoutResolver(
    @Autowired val exerciseService: ExerciseService
) {

    @DgsData(parentType = "Workout", field = "sets")
    fun exerciseSets(
        dfe: DgsDataFetchingEnvironment
    ): CompletableFuture> {
        val workout = dfe.getSource()

        // Get DataLoader and load data
        val loader = dfe.getDataLoader>("exerciseSetsLoader")
        return loader.load(workout.id)
    }
}

DataLoader Registration

In a Spring Boot application, DataLoaders are automatically registered via the `@DgsDataLoader` annotation. DGS scans the classpath and finds all such components.

Tip: DataLoaders exist only for the duration of one GraphQL request. This ensures batching works correctly, but different requests use different instances.

Subscriptions: WebSocket and Real-time

What is a Subscription?

Unlike Query (one-time request) and Mutation (change + response), Subscription is a long-lived connection where the server can send data to the client in real-time.

In Life Goals Dashboard we use Subscriptions for:

  • Achievement notifications
  • Updates about new records (workouts, meals, transactions)
  • Widget status changes

Schema for Subscription


type Subscription {
    # Subscribe to user notifications
    notificationReceived: Notification!

    # Subscribe to widget updates
    widgetDataUpdated(widgetId: ID!): WidgetData!
}

type Notification {
    id: ID!
    type: NotificationType!
    title: String!
    body: String!
    createdAt: DateTime!
}

enum NotificationType {
    ACHIEVEMENT
    REMINDER
    WARNING
    MILESTONE
}

NotificationPublisher (Redis pub/sub backed)


import org.springframework.data.redis.core.StringRedisTemplate
import org.springframework.stereotype.Service
import reactor.core.publisher.Flux
import reactor.core.publisher.Sinks
import com.mrurec.lifegoals.common.auth.ViewerContext
import com.mrurec.lifegoals.notification.model.Notification

@Service
class NotificationPublisher(
    val redisTemplate: StringRedisTemplate
) {

    private val sinks = mutableMapOf>()

    fun publishNotification(userId: String, notification: Notification) {
        // Publish to Redis channel
        val channel = "lifegoals:notifications:$userId"
        redisTemplate.convertAndSend(channel, notification)

        // Also send to Flux for WebSocket connections
        sinks[userId]?.tryEmitNext(notification)
    }

    fun getNotificationStream(): Flux {
        // Create Flux that sends notifications
        val sink = Sinks.many().multicast().onBackpressureBuffer()
        // Code simplified; in reality you need Redis channel subscription
        return sink.asFlux()
    }
}

Best Practices: Error Handling and Validation

Principle: Errors in Payload, not in GraphQL errors

In Meta systems (and in this project), business logic errors are returned in the `Payload` object, not thrown as GraphQL errors. This allows the frontend to handle errors elegantly. 


# CORRECT
type LogWorkoutPayload {
    workout: Workout
    errors: [UserError!]!
}

type UserError {
    field: String
    message: String!
    code: ErrorCode!
}

enum ErrorCode {
    VALIDATION_ERROR
    NOT_FOUND
    UNAUTHORIZED
    INTERNAL_ERROR
}

The frontend checks `payload.errors` and displays them: 


const [logWorkout, isLoading] = useMutation(LogWorkoutMutation);

const handleLogWorkout = async (input) => {
    const response = await logWorkout({ variables: { input } });

    if (response.data.logWorkout.errors.length > 0) {
        // Show errors to user
        response.data.logWorkout.errors.forEach(err => {
            console.error(err.message);
        });
    } else {
        // Workout logged successfully
        console.log("Workout logged:", response.data.logWorkout.workout);
    }
};

Server-side Validation


@DgsMutation
fun createRecipe(
    dfe: DgsDataFetchingEnvironment,
    input: CreateRecipeInput
): CreateRecipePayload {
    val viewer = dfe.getLocalContext()
    val errors = mutableListOf()

    // Validation 1: name is not empty
    if (input.name.trim().isEmpty()) {
        errors.add(UserError(
            field = "name",
            message = "Recipe name cannot be empty",
            code = ErrorCode.VALIDATION_ERROR
        ))
    }

    // Validation 2: at least 1 ingredient
    if (input.ingredients.isEmpty()) {
        errors.add(UserError(
            field = "ingredients",
            message = "Recipe must have at least one ingredient",
            code = ErrorCode.VALIDATION_ERROR
        ))
    }

    // Validation 3: check ingredient existence
    val invalidIngredients = input.ingredients.filter {
        !recipeService.productExists(viewer.userId, it.productId)
    }
    if (invalidIngredients.isNotEmpty()) {
        errors.add(UserError(
            field = "ingredients",
            message = "Some products not found",
            code = ErrorCode.NOT_FOUND
        ))
    }

    if (errors.isNotEmpty()) {
        return CreateRecipePayload(recipe = null, errors = errors)
    }

    // Create recipe if validation passed
    val recipe = recipeService.createRecipe(viewer.userId, input)
    return CreateRecipePayload(recipe = recipe, errors = emptyList())
}

                "Stack traces should not be exposed to clients")
        }
    }
}
