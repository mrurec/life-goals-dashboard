Введение: Spring Data JPA + Hibernate

JPA (Java Persistence API) — это стандарт для работы с реляционными БД через объектно-ориентированные модели. Hibernate — реализация JPA, которая автоматически преобразует объекты (entities) в таблицы и обратно. Spring Data JPA упрощает работу с Hibernate через Repository паттерн.

За 3 дня вы освоите Entity mapping (аннотации), relationships (OneToMany, ManyToOne), Repository pattern с query методами, Transactions и JSONB колонны для config.

Совет: JPA превращает ваши Kotlin классы в таблицы БД. Но помните: Hibernate генерирует SQL, который может быть неоптимальным. Всегда проверяйте логированный SQL в dev режиме (spring.jpa.show-sql=true).

Introduction: Spring Data JPA + Hibernate

JPA (Java Persistence API) is a standard for working with relational databases through object-oriented models. Hibernate is a JPA implementation that automatically converts objects (entities) into database tables and vice versa. Spring Data JPA simplifies working with Hibernate through the Repository pattern.

In 3 days you will master Entity mapping (annotations), relationships (OneToMany, ManyToOne), Repository pattern with query methods, Transactions, and JSONB columns for config.

Tip: JPA converts your Kotlin classes into database tables. But remember: Hibernate generates SQL, which may not be optimal. Always check the logged SQL in dev mode (spring.jpa.show-sql=true).

Relationships: OneToMany и ManyToOne

ManyToOne (Много к одному)

Много RecipeIngredient-ов относятся к одному FoodProduct. RecipeIngredient имеет внешний ключ foodProductId.

Relationships: OneToMany and ManyToOne

ManyToOne (Many to One)

Many RecipeIngredients relate to one FoodProduct. RecipeIngredient has a foreign key foodProductId. 


// FoodProduct — "родитель"
@Entity
@Table(name = "food_product")
data class FoodProduct(
    @Id
    val id: UUID = UUID.randomUUID(),
    val name: String,
    val caloriesPer100g: Int
)

// RecipeIngredient — "ребёнок"
@Entity
@Table(name = "recipe_ingredient")
data class RecipeIngredient(
    @Id
    val id: UUID = UUID.randomUUID(),

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "food_product_id", nullable = false)
    val foodProduct: FoodProduct,

    @Column(name = "portion_grams")
    val portionGrams: Double
)

OneToMany (Один ко многим)

Один Recipe содержит много RecipeIngredient-ов. Обратное направление ManyToOne.

OneToMany (One to Many)

One Recipe contains many RecipeIngredients. The reverse direction of ManyToOne. 


@Entity
@Table(name = "recipe")
data class Recipe(
    @Id
    val id: UUID = UUID.randomUUID(),

    @Column(nullable = false)
    val name: String,

    @OneToMany(
        mappedBy = "recipe",  // "recipe" — поле в RecipeIngredient
        fetch = FetchType.LAZY,  // Ленивая загрузка!
        cascade = [CascadeType.ALL],
        orphanRemoval = true
    )
    val ingredients: List<RecipeIngredient> = emptyList()
)

// RecipeIngredient изменяется:
@Entity
@Table(name = "recipe_ingredient")
data class RecipeIngredient(
    @Id
    val id: UUID = UUID.randomUUID(),

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "recipe_id", nullable = false)
    val recipe: Recipe,

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "food_product_id", nullable = false)
    val foodProduct: FoodProduct,

    val portionGrams: Double
)

FetchType.LAZY vs EAGER

Правило: ВСЕГДА используйте FetchType.LAZY для relationships. NEVER EAGER. Ленивая загрузка избегает N+1 проблем.

Объяснение:

  • LAZY (правильно): Recipe загружается, но ingredients не загружаются. Только при обращении к recipe.ingredients Hibernate выполнит отдельный SQL запрос.
  • EAGER (плохо): Hibernate при загрузке Recipe сразу загружает ВСЕ ingredients через JOIN. Если у вас 100 recipes, будут 100 JOIN'ов!

Решение N+1 проблемы: DataLoader

Вместо EAGER, используйте Spring Data DataLoader (как Relay на backend). DataLoader батчит запросы: вместо 100 отдельных SELECT'ов, один SELECT с IN (id1, id2, ..., id100).

FetchType.LAZY vs EAGER

Rule: ALWAYS use FetchType.LAZY for relationships. NEVER EAGER. Lazy loading avoids N+1 problems.

Explanation:

  • LAZY (correct): Recipe is loaded, but ingredients are not. Only when accessing recipe.ingredients will Hibernate execute a separate SQL query.
  • EAGER (bad): When loading a Recipe, Hibernate immediately loads ALL ingredients via JOIN. If you have 100 recipes, there will be 100 JOINs!

Solution to N+1 Problem: DataLoader

Instead of EAGER, use Spring Data DataLoader (like Relay on backend). DataLoader batches queries: instead of 100 separate SELECTs, one SELECT with IN (id1, id2, ..., id100). 


@DgsData(parentType = "Recipe", field = "ingredients")
fun ingredients(dfe: DgsDataFetchingEnvironment): CompletableFuture<List<RecipeIngredient>> {
    val loader = dfe.getDataLoader<UUID, List<RecipeIngredient>>("recipeIngredientLoader")
    val recipeId = dfe.getSource<Recipe>().id
    return loader.load(recipeId)
}

Repository Pattern

Создание Repository

Repository — это interface для доступа к сущностям. Spring Data автоматически генерирует реализацию.

Repository Pattern

Creating a Repository

A Repository is an interface for accessing entities. Spring Data automatically generates the implementation. 


import org.springframework.data.jpa.repository.JpaRepository
import org.springframework.data.jpa.repository.Query
import org.springframework.stereotype.Repository
import java.util.UUID

@Repository
interface FoodProductRepository : JpaRepository<FoodProduct, UUID> {
    // Spring автоматически реализует эти методы из имён!

    // Найти по userId
    fun findByUserId(userId: UUID): List<FoodProduct>

    // Найти по userId AND name
    fun findByUserIdAndName(userId: UUID, name: String): List<FoodProduct>

    // Поиск по pattern (LIKE)
    fun findByNameContainingIgnoreCase(name: String): List<FoodProduct>

    // Только кастомные продукты
    fun findByIsCustomTrue(): List<FoodProduct>

    // JPQL запрос
    @Query("""
        SELECT p FROM FoodProduct p
        WHERE p.userId = :userId
        AND LOWER(p.name) LIKE LOWER(CONCAT('%', :name, '%'))
        ORDER BY p.createdAt DESC
    """)
    fun searchByNameForUser(userId: UUID, name: String): List<FoodProduct>

    // Native SQL запрос
    @Query(nativeQuery = true, value = """
        SELECT * FROM food_product
        WHERE user_id = :userId AND calories_per_100g > :minCalories
        LIMIT :limit
    """)
    fun findHighCalorieProductsForUser(
        userId: UUID,
        minCalories: Int,
        limit: Int
    ): List<FoodProduct>
}

Cursor-based pagination (Relay style)

Вместо offset-based (page, size), используйте cursor-based pagination для production-ready API.

Cursor-based Pagination (Relay Style)

Instead of offset-based (page, size), use cursor-based pagination for production-ready API. 


interface RecipeRepository : JpaRepository<Recipe, UUID> {
    @Query("""
        SELECT r FROM Recipe r
        WHERE r.userId = :userId
        AND r.id > :cursor
        ORDER BY r.id ASC
        LIMIT :first
    """)
    fun findByUserIdAfterCursor(
        userId: UUID,
        cursor: UUID,
        first: Int
    ): List<Recipe>
}

// Использование
val pageSize = 10
val firstPage = recipeRepository.findByUserIdAfterCursor(
    userId = viewer.userId,
    cursor = UUID(0, 0),  // Начало
    first = pageSize + 1  // Запросить +1 для определения hasNextPage
)

val hasNextPage = firstPage.size > pageSize
val edges = firstPage.take(pageSize).map { recipe ->
    RecipeEdge(cursor = recipe.id.toString(), node = recipe)
}

Projections для оптимизации SELECT

Если нужны только некоторые поля, используйте Projections вместо загрузки полной сущности.

Projections for SELECT Optimization

If you only need certain fields, use Projections instead of loading the full entity. 


interface RecipeNameOnly {
    val id: UUID
    val name: String
    val servings: Int
}

interface RecipeRepository : JpaRepository<Recipe, UUID> {
    @Query("""
        SELECT r.id, r.name, r.servings
        FROM Recipe r
        WHERE r.userId = :userId
    """)
    fun findNameOnlyForUser(userId: UUID): List<RecipeNameOnly>
}

Transactions

@Transactional аннотация

@Transactional обёртывает метод в БД транзакцию. Если исключение выброшено, транзакция откатывается (rollback).

Transactions

@Transactional Annotation

@Transactional wraps a method in a database transaction. If an exception is thrown, the transaction is rolled back. 


@Service
class RecipeService(
    private val recipeRepository: RecipeRepository,
    private val ingredientRepository: RecipeIngredientRepository
) {
    // Вся транзакция откатывается, если любая операция не удалась
    @Transactional
    fun createRecipeWithIngredients(
        viewer: ViewerContext,
        input: CreateRecipeInput
    ): Recipe {
        // 1. Создать рецепт
        val recipe = Recipe(
            userId = viewer.userId,
            name = input.name,
            servings = input.servings
        )
        val savedRecipe = recipeRepository.save(recipe)

        // 2. Создать ингредиенты
        input.ingredients.forEach { ingredientInput ->
            val ingredient = RecipeIngredient(
                recipe = savedRecipe,
                foodProductId = ingredientInput.productId,
                portionGrams = ingredientInput.portionGrams
            )
            ingredientRepository.save(ingredient)
        }

        // Если ошибка ЗДЕСЬ — оба сохранения откатываются
        return savedRecipe
    }
}

readOnly optimization

Для методов, которые только читают — отметьте readOnly=true. Hibernate пропустит dirty checking и flush.

readOnly Optimization

For read-only methods, mark readOnly=true. Hibernate will skip dirty checking and flush. 


@Transactional(readOnly = true)
fun getRecipesForUser(viewer: ViewerContext): List<Recipe> {
    return recipeRepository.findByUserId(viewer.userId)
    // Hibernate не будет проверять изменения объектов
    // Быстрее, чем обычная транзакция
}

JSONB в JPA

Сохранение JSON в колонку

Widget config хранится как JSON в JSONB колонке PostgreSQL. JPA с Hibernate Types позволяет работать с JSON как с обычными объектами.

JSONB in JPA

Saving JSON to a Column

Widget config is stored as JSON in a JSONB column in PostgreSQL. JPA with Hibernate Types allows working with JSON like regular objects. 


import io.hypersistence.utils.hibernate.type.json.JsonType
import com.fasterxml.jackson.databind.JsonNode

@Entity
@Table(name = "user_dashboard_widget")
@TypeDefs(
    TypeDef(
        name = "json",
        typeClass = JsonType::class
    )
)
data class DashboardWidget(
    @Id
    val id: UUID = UUID.randomUUID(),

    @Column(nullable = false)
    val userId: UUID,

    @Column(nullable = false)
    val widgetType: WidgetType,

    // JSONB колонка, содержит конфиг
    @Type(JsonType::class)
    @Column(columnDefinition = "jsonb")
    val config: JsonNode,

    @Column(name = "created_at")
    val createdAt: Instant = Instant.now()
)

        val workout = workoutRepository.save(Workout(
            userId = userId,
            workoutDate = LocalDate.now(),
            totalDurationMinutes = 45
        ))
        exerciseSetRepository.save(ExerciseSet(
            workoutId = workout.id,
            exerciseName = "Squat",
            reps = 8,
            sets = 4,
            weightKg = 120.0
        ))

        workoutRepository.delete(workout)

        val sets = exerciseSetRepository.findByWorkoutId(workout.id)
        assertThat(sets).isEmpty()
    }

    @Test
    fun `findByUserIdAndWorkoutDateBetween should filter by date range`() {
        val today = LocalDate.now()
        workoutRepository.save(Workout(userId = userId, workoutDate = today, totalDurationMinutes = 60))
        workoutRepository.save(Workout(userId = userId, workoutDate = today.minusDays(5), totalDurationMinutes = 45))
        workoutRepository.save(Workout(userId = userId, workoutDate = today.minusDays(10), totalDurationMinutes = 50))

        val weekWorkouts = workoutRepository.findByUserIdAndWorkoutDateBetween(
            userId, today.minusDays(7), today
        )

        assertThat(weekWorkouts).hasSize(2)
    }
}

            carbsG = 5.0,
            fatG = 1.5,
            isCustom = true
        )

        val saved = repository.save(customProduct)

        assertThat(saved.userId).isEqualTo(userId)
        assertThat(saved.isCustom).isTrue()
    }

    @Test
    fun `findByNameContainingIgnoreCaseAndIsCustom should search catalog and user products`() {
        repository.save(FoodProduct(userId = null, name = "Banana", caloriesPer100g = 89, isCustom = false))
        repository.save(FoodProduct(userId = userId, name = "Banana Protein Mix", caloriesPer100g = 150, isCustom = true))
        repository.save(FoodProduct(userId = userId, name = "Apple", caloriesPer100g = 52, isCustom = false))

        val userBananaProducts = repository.findByNameContainingIgnoreCaseAndUserIdAndIsCustom("banana", userId, true)
        val catalogProducts = repository.findByNameContainingIgnoreCaseAndUserIdIsNullAndIsCustom("banana", false)

        assertThat(userBananaProducts).hasSize(1)
        assertThat(userBananaProducts[0].name).isEqualTo("Banana Protein Mix")
        assertThat(catalogProducts).hasSize(1)
    }

    @Test
    fun `countByUserIdAndIsCustom should count user products separately`() {
        repository.save(FoodProduct(userId = userId, name = "Custom 1", caloriesPer100g = 100, isCustom = true))
        repository.save(FoodProduct(userId = userId, name = "Custom 2", caloriesPer100g = 150, isCustom = true))
        repository.save(FoodProduct(userId = null, name = "Catalog 1", caloriesPer100g = 80, isCustom = false))

        val customCount = repository.countByUserIdAndIsCustom(userId, true)
        val catalogCount = repository.countByUserIdIsNullAndIsCustom(false)

        assertThat(customCount).isEqualTo(2)
        assertThat(catalogCount).isEqualTo(1)
    }
