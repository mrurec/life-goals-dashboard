Spring Boot 4 adds built-in API versioning support. Four main strategies: URL path (/v1/users), Header (API-Version: 1), Media-Type (application/vnd.app+json;version=1), Query param (?version=1). Configure via spring.mvc.apiversion.* or @RequestMapping(version="1.0") annotation.

REST Client для внешних APIREST Client for External APIs

RestClient и HttpExchange в Spring Boot 4RestClient and HttpExchange in Spring Boot 4

Spring Boot 4 имеет встроенный RestClient и декларативные @HttpExchange интерфейсы для экстерн API интеграции. Это современные альтернативы RestTemplate и WebClient, с автоматической регистрацией как бин.

Spring Boot 4 has a built-in RestClient and declarative @HttpExchange interfaces for external API integration. These are modern alternatives to RestTemplate and WebClient, with automatic bean registration. 


// Конфигурация
@Configuration
class HttpClientConfig {
    @Bean
    fun restClient(): RestClient {
        return RestClient.builder()
            .requestFactory(SimpleClientHttpRequestFactory())
            .messageConverters { converters ->
                converters.add(MappingJackson2HttpMessageConverter())
            }
            .build()
    }
}

// Использование в сервисе
@Service
class ExchangeRateApi(
    private val restClient: RestClient,
    private val properties: ExchangeRateProperties
) {
    data class RateResponse(
        val rates: Map<String, BigDecimal>
    )

    fun fetchRate(base: String, target: String): BigDecimal {
        val response = restClient.get()
            .uri("${properties.url}?access_key=${properties.key}&base=$base&symbols=$target")
            .retrieve()
            .body(RateResponse::class.java)

        return response?.rates?.get(target) ?: throw RuntimeException("Rate not found")
    }
}

Payload типыPayload Types

Каждая мутация возвращает Payload с результатом И ошибками. Никогда не кидаем исключения для бизнес-ошибок.

Each mutation returns a Payload with result AND errors. Never throw exceptions for business errors. 


// GraphQL Schema: mutations.graphqls
type CreateRecipePayload {
  recipe: Recipe
  errors: [UserError!]!
}

type UserError {
  field: String
  message: String!
  code: ErrorCode!
}

enum ErrorCode {
  VALIDATION_ERROR
  NOT_FOUND
  INTERNAL_ERROR
}

Реализация в Kotlin:Implementation in Kotlin:


@Service
class RecipeService(
    private val repository: RecipeRepository,
    private val productService: FoodProductService
) {
    fun createRecipe(input: CreateRecipeInput): CreateRecipePayload {
        val errors = mutableListOf<UserError>()

        // Валидация имени
        if (input.name.isBlank()) {
            errors.add(UserError(
                field = "name",
                message = "Имя рецепта не может быть пустым",
                code = ErrorCode.VALIDATION_ERROR
            ))
        }

        // Валидация ингредиентов
        if (input.ingredients.isEmpty()) {
            errors.add(UserError(
                field = "ingredients",
                message = "Рецепт должен содержать хотя бы один ингредиент",
                code = ErrorCode.VALIDATION_ERROR
            ))
        }

        // Если ошибки валидации — вернуть их в Payload
        if (errors.isNotEmpty()) {
            return CreateRecipePayload(recipe = null, errors = errors)
        }

        // Проверить существование продуктов
        val products = input.ingredients.mapNotNull { ingredient ->
            val product = repository.findById(ingredient.productId)
            if (product.isEmpty) {
                errors.add(UserError(
                    field = "ingredients[${ingredient.productId}]",
                    message = "Продукт не найден",
                    code = ErrorCode.NOT_FOUND
                ))
                null
            } else {
                product.get()
            }
        }

        if (errors.isNotEmpty()) {
            return CreateRecipePayload(recipe = null, errors = errors)
        }

        // Создать рецепт
        val recipe = Recipe(
            name = input.name,
            ingredients = products,
            servings = input.servings
        )
        val saved = repository.save(recipe)

        return CreateRecipePayload(recipe = saved, errors = emptyList())
    }
}

data class CreateRecipePayload(
    val recipe: Recipe?,
    val errors: List<UserError>
)

@Service
class WorkoutService(
    private val repository: WorkoutRepository
) {
    fun getMyWorkouts(viewer: ViewerContext): List<Workout> {
        // Получить только тренировки ТЕКУЩЕГО пользователя
        return repository.findByUserId(viewer.userId)
    }

    fun deleteWorkout(viewer: ViewerContext, workoutId: UUID) {
        val workout = repository.findById(workoutId)
            .orElseThrow { NotFoundException("Workout not found") }

        // Проверка: может ли viewer удалить эту тренировку?
        if (workout.userId != viewer.userId) {
            throw ForbiddenException("You can only delete your own workouts")
        }

        repository.delete(workout)
    }
}

Spring Modulith: энфорсмент границ модульного монолитаSpring Modulith: Enforcing Modular Monolith Boundaries

Проект — это widget-first modular monolith (см. CLAUDE.md → Architecture). Каждый пакет верхнего уровня (interviewprep, fitness, budget, dashboard, notification, common) — модуль со строгими границами. Виджеты могут зависеть только от common — никогда напрямую от repository/ или model/ друг друга.

Spring Modulith — инструмент, который энфорсит это правило в CI, а не в code review. Полное обоснование выбора (и почему не ArchUnit) — в docs/adr/ADR-004-module-boundary-enforcement.md.

Почему нельзя просто договориться?

  • Code review не масштабируется. Даже на проекте из 1 человека уставший пятничный merge рано или поздно пересечёт границу. CI ловит это в 100% случаев.
  • Нет drift документации. Modulith генерирует PlantUML-диаграммы из самого кода — диаграмма физически не может разойтись с реальностью.
  • События заменяют скрытые связи. Без формальной event-шины cross-module фичи (например, notification реагирует на fitness.WorkoutLogged) молча превращаются в прямые вызовы.

The project is a widget-first modular monolith (see CLAUDE.md → Architecture). Every top-level package (interviewprep, fitness, budget, dashboard, notification, common) is a module with explicit boundaries. Widgets may depend only on common — never on each other's repository/ or model/.

Spring Modulith is the tool that enforces this in CI, not by code review. The full decision (and why not ArchUnit) is in docs/adr/ADR-004-module-boundary-enforcement.md.

Why not just discipline?

  • Code review doesn't scale. Even on a 1-person project, a tired Friday merge will cross a boundary. CI catches it 100% of the time.
  • No documentation drift. Modulith generates PlantUML diagrams from the code itself — the diagram can never diverge from reality.
  • Events replace hidden coupling. Without a formal event bus, cross-module features (e.g., notification reacting to fitness.WorkoutLogged) silently become direct calls.

Cross-module событияCross-Module Events

Никаких прямых зависимостей между модулями — только через события:

No direct dependencies between modules — only events:

// fitness публикует — ничего не знает о слушателях
data class WorkoutLogged(val userId: UUID, val workoutId: UUID, val at: Instant)

@Service
class WorkoutService(private val events: ApplicationEventPublisher) {
    @Transactional
    fun logWorkout(/* ... */) {
        // ... persist workout ...
        events.publishEvent(WorkoutLogged(viewer.userId, workout.id, Instant.now()))
    }
}

// notification реагирует — нулевая compile-time зависимость от fitness
@Component
class WorkoutAchievementListener(
    private val notifications: NotificationService,
    private val achievements: AchievementChecker
) {
    @ApplicationModuleListener    // = @TransactionalEventListener + @Async + persistence
    fun on(event: WorkoutLogged) {
        val earned = achievements.check(event.userId)
        earned.forEach { notifications.publish(event.userId, it) }
    }
}

@ApplicationModuleListener — ключевая штука: событие сохраняется в таблице event_publication внутри той же транзакции, что записала workout. Если слушатель упал или приложение крашнулось — Modulith ретраит при рестарте. Это transactional outbox pattern — тот же подход, что Meta/LinkedIn используют для надёжных cross-service записей, только без Kafka посередине.

@ApplicationModuleListener is key: the event is persisted in the event_publication table inside the same transaction that wrote the workout. If the listener fails or the app crashes, Modulith retries on restart. This is the transactional outbox pattern — the same approach Meta/LinkedIn use for reliable cross-service writes, but without Kafka in the way.

This runs ~10× faster than full @SpringBootTest and proves the module works in isolation — the contract it exposes and the events it publishes are all that matter for its callers.

Trade-off vs ArchUnitTrade-off vs ArchUnit

ArchUnit — library-agnostic и максимально гибкий, любое правило выразимо в DSL. Modulith приходит с готовой моделью модульного монолита: знает, что такое модуль, цикл, и public API. Для нашего набора правил Modulith покрывает всё и даёт per-module тесты, события и авто-документацию сверху. ArchUnit оставлен в резерве для редких правил, которые Modulith не выражает. Полное обоснование — в ADR-004.

ArchUnit is library-agnostic and maximally flexible — any rule expressible in DSL. Modulith ships with the modular-monolith model built in: it knows what a module is, what a cycle is, and what public API means. For our rule set Modulith covers everything and adds per-module tests, events, and auto-docs on top. ArchUnit is kept in reserve for rare rules Modulith cannot express. Full decision in ADR-004.

FAANG-interview framingFAANG-Interview Framing

На вопрос «Why this architecture?» — структурированный ответ:

  • Enforcement поверх convention — senior-level judgment.
  • Transactional event publication — то же семейство, что Meta TAO write pipeline и LinkedIn outbox pattern.
  • Per-module тесты — аналог service-level изоляции в микросервисной организации.
  • Документация-как-код — никакого Confluence drift.
  • Явный путь извлечения — граница каждого модуля уже соответствует будущему service-контракту; локальные listener'ы становятся Kafka consumer'ами без изменения API.

When asked «Why this architecture?» — a structured answer:

  • Enforcement over convention — senior-level judgment.
  • Transactional event publication — same family as Meta's TAO write pipeline and LinkedIn's outbox pattern.
  • Per-module tests — analogous to service-level isolation in a microservice org.
  • Documentation as code — no Confluence drift.
  • Clear extraction path — each module's boundary already matches a future service contract; local listeners become Kafka consumers with zero API change.
