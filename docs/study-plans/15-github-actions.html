Введение: Автоматизация через GitHub Actions

GitHub Actions — это встроенная система CI/CD (Continuous Integration / Continuous Deployment) в GitHub. Вы определяете workflow в YAML файлах, и GitHub автоматически запускает их на каждый push, pull request, или по расписанию.

Для Life Goals Dashboard мы используем Actions для: - Тестирования кода при каждом PR - Линтирования и проверки типов - Развёртывания на Railway (backend) и Vercel (frontend)

Совет: За 1.5 дня вы создадите полный CI workflow для backend и frontend, плюс deploy workflow.

Workflow файлы: основы

Структура


.github/
└── workflows/
    ├── ci.yml              # Тестирование на push и PR
    └── deploy.yml          # Развёртывание (production)

Базовая структура YAML workflow


name: CI                    # Имя workflow

on:                         # Когда запускать
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:                     # Имя job
    runs-on: ubuntu-latest  # Runner (машина)
    steps:
      - uses: actions/checkout@v4
      - run: echo "Hello"

Ключевые концепции

Концепция Описание
Workflow YAML файл, который определяет automated process
Event Когда запускается workflow (push, pull_request, schedule)
Job Набор steps, которые выполняются на одной машине
Step Отдельная команда или action
Runner Машина, где выполняется job (ubuntu-latest, macos-latest, windows-latest)

Backend CI Job: Kotlin + Gradle

Полный backend CI workflow


name: Backend CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest

    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_PASSWORD: postgres
          POSTGRES_DB: lifegoals
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432

      redis:
        image: redis:7-alpine
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 6379:6379

    steps:
      # 1. Checkout код
      - uses: actions/checkout@v4

      # 2. Установка JDK
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
          cache: gradle

      # 3. Lint
      - name: Kotlin Lint (ktlint)
        run: ./gradlew ktlintCheck

      # 4. Тесты
      - name: Run Tests
        run: ./gradlew test
        env:
          DB_URL: jdbc:postgresql://localhost:5432/lifegoals
          DB_USERNAME: postgres
          DB_PASSWORD: postgres
          REDIS_URL: redis://localhost:6379

      # 5. Build
      - name: Build JAR
        run: ./gradlew build -x test

Расшифровка backend job

Step Что делает
actions/checkout@v4 Скачивает код из репо
actions/setup-java@v4 Устанавливает JDK 21
cache: gradle Кеширует Gradle зависимости (ускорение)
services Запускает PostgreSQL и Redis контейнеры для тестов
--health-cmd Проверяет, готов ли сервис (healthcheck)

Frontend CI Job: Node + npm

Полный frontend CI workflow


name: Frontend CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [18, 20]  # Тестируем на нескольких версиях Node

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: npm
          cache-dependency-path: life-goals-web/package-lock.json

      - name: Install Dependencies
        working-directory: life-goals-web
        run: npm ci

      - name: Relay Compiler
        working-directory: life-goals-web
        run: npm run relay

      - name: TypeScript Check
        working-directory: life-goals-web
        run: npm run typecheck

      - name: ESLint
        working-directory: life-goals-web
        run: npm run lint

      - name: Unit Tests
        working-directory: life-goals-web
        run: npm test -- --coverage

      - name: Build Bundle
        working-directory: life-goals-web
        run: npm run build

Параметры для frontend

Параметр Значение Описание
working-directory life-goals-web Папка фронтенда (если не в корне)
cache: npm — Кеширует node_modules
npm ci вместо npm install Детерминированная установка (для CI)
strategy.matrix node-version: [18, 20] Запускает job на каждой версии

Deploy Workflow: Production

Deploy workflow с dependencies


name: Deploy

on:
  push:
    branches: [main]

jobs:
  ci:
    uses: ./.github/workflows/ci.yml  # Перепроверяем тесты перед deploy

  deploy-backend:
    needs: ci                         # Ждём CI job
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Deploy to Railway
        run: |
          npx railway link ${{ secrets.RAILWAY_PROJECT_ID }}
          npx railway deploy --token ${{ secrets.RAILWAY_API_TOKEN }}

  deploy-frontend:
    needs: ci
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: npm
          cache-dependency-path: life-goals-web/package-lock.json

      - name: Install & Build
        working-directory: life-goals-web
        run: |
          npm ci
          npm run relay
          npm run build

      - name: Deploy to Vercel
        uses: vercel/action@master
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}

Secrets в GitHub

Secrets хранятся в Settings → Secrets and variables → Actions. Обращайтесь через ${{ secrets.SECRET_NAME }}.

Secret Для
RAILWAY_API_TOKEN Railway deployment API
VERCEL_TOKEN Vercel deployment
VERCEL_ORG_ID Vercel organization
VERCEL_PROJECT_ID Vercel project
Важно: Никогда не коммитьте secrets в код! Используйте GitHub Secrets.

Backlog #4: GitHub Actions CI/CD pipeline 5 SP

Описание: Создайте GitHub Actions workflow который при каждом push'е на main: запускает gradle build и тесты для backend, затем npm build + tests для frontend. На успех — собирает Docker image и пушит на Container Registry.

TDD тест (bash скрипт):

#!/bin/bash
# tests/github-actions.sh

set -e

echo "Testing GitHub Actions workflow..."

# Проверка что файл workflow существует
if [ ! -f ".github/workflows/ci.yml" ]; then
    echo "ERROR: .github/workflows/ci.yml not found"
    exit 1
fi

WORKFLOW_FILE=".github/workflows/ci.yml"

# Проверка название job'а
echo "Checking workflow jobs..."
HAS_BACKEND_JOB=$(grep -c "backend:" "$WORKFLOW_FILE" || true)
HAS_FRONTEND_JOB=$(grep -c "frontend:" "$WORKFLOW_FILE" || true)

if [ "$HAS_BACKEND_JOB" -eq 0 ]; then
    echo "ERROR: Backend job not defined"
    exit 1
fi

if [ "$HAS_FRONTEND_JOB" -eq 0 ]; then
    echo "ERROR: Frontend job not defined"
    exit 1
fi

# Проверка на checkout step
echo "Checking for checkout action..."
HAS_CHECKOUT=$(grep -c "actions/checkout" "$WORKFLOW_FILE" || true)
if [ "$HAS_CHECKOUT" -eq 0 ]; then
    echo "ERROR: Checkout action not found"
    exit 1
fi

# Проверка на setup-java
echo "Checking for Java setup..."
HAS_JAVA=$(grep -c "setup-java" "$WORKFLOW_FILE" || true)
if [ "$HAS_JAVA" -eq 0 ]; then
    echo "ERROR: Java setup not found"
    exit 1
fi

# Проверка на setup-node
echo "Checking for Node.js setup..."
HAS_NODE=$(grep -c "setup-node" "$WORKFLOW_FILE" || true)
if [ "$HAS_NODE" -eq 0 ]; then
    echo "ERROR: Node.js setup not found"
    exit 1
fi

# Проверка backend steps
echo "Checking backend build steps..."
HAS_GRADLE_BUILD=$(grep -c "gradlew build" "$WORKFLOW_FILE" || true)
HAS_GRADLE_TEST=$(grep -c "gradlew test" "$WORKFLOW_FILE" || true)

if [ "$HAS_GRADLE_BUILD" -eq 0 ]; then
    echo "ERROR: Gradle build step not found"
    exit 1
fi

if [ "$HAS_GRADLE_TEST" -eq 0 ]; then
    echo "ERROR: Gradle test step not found"
    exit 1
fi

# Проверка frontend steps
echo "Checking frontend build steps..."
HAS_NPM_CI=$(grep -c "npm ci" "$WORKFLOW_FILE" || true)
HAS_NPM_BUILD=$(grep -c "npm run build" "$WORKFLOW_FILE" || true)
HAS_NPM_TEST=$(grep -c "npm test" "$WORKFLOW_FILE" || true)

if [ "$HAS_NPM_CI" -eq 0 ]; then
    echo "ERROR: npm ci step not found"
    exit 1
fi

if [ "$HAS_NPM_BUILD" -eq 0 ]; then
    echo "ERROR: npm build step not found"
    exit 1
fi

if [ "$HAS_NPM_TEST" -eq 0 ]; then
    echo "ERROR: npm test step not found"
    exit 1
fi

# Проверка Docker image push
echo "Checking Docker image steps..."
HAS_DOCKER_BUILD=$(grep -c "build-push-action" "$WORKFLOW_FILE" || true)

if [ "$HAS_DOCKER_BUILD" -eq 0 ]; then
    echo "ERROR: Docker build-push action not found"
    exit 1
fi

# Проверка trigger на main branch
echo "Checking workflow trigger..."
HAS_MAIN_TRIGGER=$(grep -c "main" "$WORKFLOW_FILE" || true)

if [ "$HAS_MAIN_TRIGGER" -eq 0 ]; then
    echo "ERROR: Main branch trigger not found"
    exit 1
fi

echo "SUCCESS: All workflow checks passed"
exit 0

Introduction: Automation with GitHub Actions

GitHub Actions is a built-in CI/CD (Continuous Integration / Continuous Deployment) system in GitHub. You define workflows in YAML files, and GitHub automatically runs them on every push, pull request, or on a schedule.

For Life Goals Dashboard, we use Actions for: - Testing code on every PR - Linting and type checking - Deploying to Railway (backend) and Vercel (frontend)

Tip: In 1.5 days you will create a complete CI workflow for backend and frontend, plus a deploy workflow.

Workflow Files: Basics

Structure


.github/
└── workflows/
    ├── ci.yml              # Testing on push and PR
    └── deploy.yml          # Deployment (production)

Basic YAML Workflow Structure


name: CI                    # Workflow name

on:                         # When to run
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:                     # Job name
    runs-on: ubuntu-latest  # Runner (machine)
    steps:
      - uses: actions/checkout@v4
      - run: echo "Hello"

Key Concepts

Concept Description
Workflow YAML file that defines an automated process
Event When the workflow runs (push, pull_request, schedule)
Job Set of steps that run on a single machine
Step Individual command or action
Runner Machine where the job runs (ubuntu-latest, macos-latest, windows-latest)
