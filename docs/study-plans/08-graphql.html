Введение: GraphQL для Life Goals DashboardIntroduction: GraphQL for Life Goals Dashboard

GraphQL — это язык запросов для API. Клиент запрашивает РОВНО те данные, которые ему нужны, на одном endpoint. Нет overfetching (лишние данные) и underfetching (нужны доп. запросы).

GraphQL используется на всех крупных компаниях (Meta, GitHub, Shopify, Twitter). Life Goals Dashboard использует Netflix DGS (Domain Graph Service) для реализации GraphQL на Spring Boot.

За 2 дня вы освоите GraphQL schema, типы, queries, mutations, subscriptions, Relay Connection pattern и directives (@defer для оптимизации).

Совет: GraphQL можно тестировать в GraphiQL IDE (встроена в DGS на :8080/graphiql). Пишите запросы прямо в браузере и видите результаты в реальном времени.

GraphQL is a query language for APIs. A client requests exactly the data it needs, on a single endpoint. No overfetching (unnecessary data) and no underfetching (need for additional requests).

GraphQL is used by major companies (Meta, GitHub, Shopify, Twitter). Life Goals Dashboard uses Netflix DGS (Domain Graph Service) to implement GraphQL on Spring Boot.

In 2 days, you will master GraphQL schema, types, queries, mutations, subscriptions, Relay Connection pattern, and directives (@defer for optimization).

Tip: GraphQL can be tested in GraphiQL IDE (built-in to DGS at :8080/graphiql). Write queries directly in the browser and see results in real-time.

GraphQL vs RESTGraphQL vs REST

Проблемы RESTREST Problems

Overfetching: Клиент запрашивает `GET /users/123` и получает все поля пользователя, даже если нужны только имя и email.

Underfetching: Клиент запрашивает `GET /users/123`, получает пользователя, потом делает `GET /users/123/workouts` для тренировок. Два запроса вместо одного.

Overfetching: Client requests `GET /users/123` and receives all user fields, even if only name and email are needed.

Underfetching: Client requests `GET /users/123`, gets the user, then makes `GET /users/123/workouts` for workouts. Two requests instead of one.

Решение: GraphQLSolution: GraphQL


# Client requests exactly what it needs
query {
  me {
    id
    name
    email
    workouts {
      id
      date
      durationMinutes
    }
  }
}

# Response contains exactly these fields, nothing extra
{
  "me": {
    "id": "uuid-123",
    "name": "mrurec",
    "email": "yura@example.com",
    "workouts": [
      {
        "id": "workout-1",
        "date": "2026-04-13",
        "durationMinutes": 45
      }
    ]
  }
}

Schema и TypesSchema and Types

Что такое Schema?What is Schema?

GraphQL Schema — это контракт между клиентом и сервером. Он определяет: - Какие данные доступны (types) - Какие операции возможны (Query, Mutation, Subscription) - Какие поля имеют какие типы

GraphQL Schema is a contract between client and server. It defines: - What data is available (types) - What operations are possible (Query, Mutation, Subscription) - What types fields have

Основные типыBasic Types


# Scalar types (built-in)
scalar String
scalar Int
scalar Float
scalar Boolean
scalar ID          # Unique identifier
scalar DateTime    # ISO 8601 date-time
scalar Date        # Date (without time)
scalar JSON        # Arbitrary JSON (e.g., widget config)

# Object type — entity with fields
type User {
  id: ID!                    # ! means NOT NULL (required)

  workouts: [Workout!]!      # Array [Type!] — array of non-null elements

interface Node {
  id: ID!
}

type Workout implements Node {
  id: ID!
  userId: ID!

  caloriesBurned: Int
}

# Enum — enumeration of values
enum MealType {
  BREAKFAST

  SNACK
}

type Meal {
  id: ID!

union WidgetData = FitnessWidgetData | InterviewPrepWidgetData | BudgetWidgetData

Input типы (для мутаций)Input Types (for Mutations)

Input типы — это объекты, которые клиент ПЕРЕДАЁТ серверу (в мутациях). Output типы — это объекты, которые сервер ВОЗВРАЩАЕТ клиенту.

Input types are objects that the client SENDS to the server (in mutations). Output types are objects that the server RETURNS to the client. 


input CreateWorkoutInput {
  date: Date!
  durationMinutes: Int!
  caloriesBurned: Int
}

type CreateWorkoutPayload {
  workout: Workout
  errors: [UserError!]!
}

type UserError {
  field: String
  message: String!
  code: ErrorCode!
}

enum ErrorCode {
  VALIDATION_ERROR
  UNAUTHORIZED
  NOT_FOUND
  INTERNAL_ERROR
}

Queries и MutationsQueries and Mutations

Query — чтение данныхQuery — Reading Data

Query — это "GET" запрос. Не меняет данные, только читает.

Query is a "GET" request. Does not modify data, only reads. 


type Query {
  # Get current user
  me: User!

  # Get user by ID
  user(id: ID!): User

  # Get all recipes of current user
  myRecipes: [Recipe!]!

  # Search with parameters
  searchRecipes(query: String!, limit: Int = 10): [Recipe!]!
}

# Examples of queries from client:
query {
  me {
    id
    name
    email
  }
}

query {
  myRecipes {
    id
    name
    servings
    ingredients {
      id
      product {
        name
        caloriesPer100g
      }
      portionGrams
    }
  }
}

query SearchRecipes($query: String!) {
  searchRecipes(query: $query, limit: 20) {
    id
    name
  }
}

Mutation — изменение данныхMutation — Modifying Data

Mutation — это "POST/PUT/DELETE" запрос. Изменяет данные и возвращает новое состояние. ВСЕГДА возвращает Payload с errors массивом (никогда не кидаем исключения для бизнес-ошибок).

Mutation is a "POST/PUT/DELETE" request. Modifies data and returns new state. ALWAYS returns a Payload with errors array (never throw exceptions for business errors). 


type Mutation {
  # Log a workout
  logWorkout(input: LogWorkoutInput!): LogWorkoutPayload!

  # Create recipe
  createRecipe(input: CreateRecipeInput!): CreateRecipePayload!

  # Log meal
  logMeal(input: LogMealInput!): LogMealPayload!
}

input LogWorkoutInput {
  date: Date!
  durationMinutes: Int!
  caloriesBurned: Int
  notes: String
}

type LogWorkoutPayload {
  workout: Workout
  errors: [UserError!]!
}

# Example mutation from client:
mutation LogWorkout($input: LogWorkoutInput!) {
  logWorkout(input: $input) {
    workout {
      id
      date
      durationMinutes
    }
    errors {
      field
      message
      code
    }
  }
}

Subscriptions (WebSocket)Subscriptions (WebSocket)

Real-time уведомленияReal-time Notifications

Subscription — это "слушатель" на данные. Клиент подписывается через WebSocket, и сервер отправляет данные в реальном времени.

Subscription is a "listener" for data. Client subscribes via WebSocket, and server sends data in real-time. 


type Subscription {
  # Listen to notifications of current user
  myNotifications: Notification!

  # Listen to recipe updates in real-time
  recipeUpdated(recipeId: ID!): Recipe!
}

type Notification {
  id: ID!
  type: NotificationType!
  message: String!
  timestamp: DateTime!
}

enum NotificationType {
  ACHIEVEMENT_UNLOCKED
  REMINDER
  MILESTONE_REACHED
  WARNING
}

# Example subscription from client:
subscription {
  myNotifications {
    id
    type
    message
    timestamp
  }
}

# If server publishes an event via Redis pub/sub,
# client receives it IMMEDIATELY via WebSocket:
{
  "myNotifications": {
    "id": "notif-123",
    "type": "ACHIEVEMENT_UNLOCKED",
    "message": "3 workouts this week!",
    "timestamp": "2026-04-13T10:30:00Z"
  }
}

Relay Connection PatternRelay Connection Pattern

Курсорная пагинацияCursor-based Pagination

Relay Connection pattern — это стандарт для пагинации в GraphQL. Используется cursor-based pagination (вместо offset-based), что позволяет безопасно работать с изменяющимися данными.

Relay Connection pattern is a standard for pagination in GraphQL. Uses cursor-based pagination (instead of offset-based), which allows safe work with changing data. 


# Relay types (standard for any API)
interface Node {
  id: ID!
}

type PageInfo {
  hasNextPage: Boolean!
  hasPreviousPage: Boolean!
  startCursor: String
  endCursor: String
}

# Connection and Edge types for any type
type RecipeConnection {
  edges: [RecipeEdge!]!
  pageInfo: PageInfo!
  totalCount: Int!
}

type RecipeEdge {
  cursor: String!          # Unique identifier of row in list
  node: Recipe!            # The entity itself (Recipe)
}

# Now Query uses Connection types
type Query {
  myRecipes(
    first: Int                  # How many elements to take (forward)
    after: String              # After which cursor to start
    last: Int                  # How many elements to take (backward)
    before: String             # Before which cursor to take
  ): RecipeConnection!
}

# Client request:
query MyRecipes($first: Int!, $after: String) {
  myRecipes(first: $first, after: $after) {
    edges {
      cursor
      node {
        id
        name
      }
    }
    pageInfo {
      hasNextPage
      endCursor
    }
    totalCount
  }
}

# Response:
{
  "myRecipes": {
    "edges": [
      {
        "cursor": "recipe:uuid1",
        "node": { "id": "uuid1", "name": "Pasta" }
      },
      {
        "cursor": "recipe:uuid2",
        "node": { "id": "uuid2", "name": "Salad" }
      }
    ],
    "pageInfo": {
      "hasNextPage": true,
      "endCursor": "recipe:uuid2"
    },
    "totalCount": 42
  }
}

# Next page:
query NextRecipes($first: Int!, $after: String!) {
  myRecipes(first: $first, after: $after) {
    # ...
  }
}
# Passed variables: { "first": 10, "after": "recipe:uuid2" }

DirectivesDirectives

@defer — отложенная загрузка@defer — Deferred Loading

@defer директива говорит серверу: "Отправь это поле позже, не ждите для основного ответа". Используется для тяжёлых полей (которые долго вычисляются).

@defer directive tells server: "Send this field later, don't wait for the main response". Used for heavy fields (that take long to compute). 


# In schema we define @defer as possible:
directive @defer(label: String, if: Boolean = true) on FIELD | FRAGMENT_SPREAD

# In query:
query DashboardPage {
  myDashboard {
    id
    widgets {
      id
      type
      position

      # Server sends this data after main response
      data @defer(label: "widget_data") {
        __typename
        ... on FitnessWidgetData {
          currentWeight
          targetWeight
        }
        ... on BudgetWidgetData {
          saved
          target
        }
      }
    }
  }
}

# Client receives response in 2 parts:
# 1. Immediately (200ms):
{
  "myDashboard": {
    "widgets": [
      { "id": "1", "type": "FITNESS", "position": 0 }
    ]
  }
}

# 2. Later (1s), in separate chunk:
{
  "label": "widget_data",
  "data": {
    "__typename": "FitnessWidgetData",
    "currentWeight": 82.5,
    "targetWeight": 80
  }
}

Fragments — переиспользуемые блоки запросаFragments — Reusable Query Blocks

Fragment — это переиспользуемый блок полей. Колокируется с компонентом (в одном файле).

Fragment is a reusable block of fields. Colocated with component (in the same file). 


# Fragment definition
fragment RecipeCard_recipe on Recipe {
  id
  name
  servings
  ingredients {
    id
    product {
      name
    }
  }
}

# Fragment usage
query MyRecipes {
  myRecipes {
    ...RecipeCard_recipe
  }
}

Create schema with types: Transaction, SavingsGoal, BudgetWidgetData, ExchangeRate (cached). Mutation logTransaction with validation. Query getExchangeRate(from, to) with Redis caching. Subscription onTransactionLogged for real-time updates.

Test:


@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class BudgetSchemaTest {
    @Autowired
    private lateinit var graphQLTester: GraphQLTester

    @Test
    fun `logTransaction mutation should validate amount and currency`() {
        val mutation = """
            mutation LogTransaction(${'$'}input: LogTransactionInput!) {
              logTransaction(input: ${'$'}input) {
                transaction {
                  id
                  amount
                  currency
                  description
                  transactionDate
                }
                errors {
                  field
                  message
                  code
                }
              }
            }
        """
        val variables = mapOf(
            "input" to mapOf(
                "amount" to 150.50,
                "currency" to "EUR",
                "description" to "Flight to South Korea",
                "transactionDate" to "2026-04-13"
            )
        )

        val result = graphQLTester.executeQuery(mutation, variables)

        assertThat(result.path("logTransaction.transaction.id")).isNotNull()
        assertThat(result.path("logTransaction.transaction.amount")).isEqualTo(150.50)
    }

    @Test
    fun `getExchangeRate should return rate and be cached`() {
        val query = """
            query GetRate(${'$'}from: String!, ${'$'}to: String!) {
              getExchangeRate(from: ${'$'}from, to: ${'$'}to) {
                from
                to
                rate
                timestamp
              }
            }
        """
        val variables = mapOf("from" to "EUR", "to" to "KRW")

        val result1 = graphQLTester.executeQuery(query, variables)
        val result2 = graphQLTester.executeQuery(query, variables)

        assertThat(result1.path("getExchangeRate.rate")).isNotNull()
        assertThat(result1.path("getExchangeRate.rate")).isEqualTo(result2.path("getExchangeRate.rate"))
    }

    @Test
    fun `BudgetWidgetData should return progress towards goal`() {
        val query = """
            query {
              myDashboard {
                id
                widgets {
                  ... on BudgetWidgetData {
                    goalName
                    saved
                    target
                    targetCurrency
                    deadline
                    savingsForecast {
                      projectedReachDate
                      scenarios {
                        label
                        projectedAmount
                      }
                    }
                  }
                }
              }
            }
        """

        val result = graphQLTester.executeQuery(query)

        assertThat(result.path("myDashboard.widgets[0].goalName")).isNotNull()
        assertThat(result.path("myDashboard.widgets[0].savingsForecast.scenarios")).isInstanceOf(List::class.java)
    }
}
