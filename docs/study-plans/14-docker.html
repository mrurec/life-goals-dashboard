Введение: Контейнеризация для Life Goals Dashboard

Docker позволяет упаковать приложение с его зависимостями в контейнер, который работает одинаково везде: на вашем ноутбуке, на сервере CI/CD, в продакшене.

Docker Compose позволяет запустить несколько контейнеров (API, PostgreSQL, Redis) одной командой, с определением сетей и томов.

Совет: За 2 дня вы создадите Dockerfile для Spring Boot приложения и docker-compose.yml для полного локального окружения.
JDK 25 LTS: Мы используем JDK 25 LTS (сентябрь 2025) — минимум Spring Boot 4 требует Java 17, но мы идём на LTS-25 для долгосрочной поддержки.

Introduction: Containerization for Life Goals Dashboard

Docker allows you to package an application with its dependencies in a container, which works identically everywhere: on your laptop, on CI/CD servers, and in production.

Docker Compose lets you run multiple containers (API, PostgreSQL, Redis) with a single command, defining networks and volumes.

Tip: In 2 days, you will create a Dockerfile for a Spring Boot application and docker-compose.yml for a complete local environment.
JDK 25 LTS: We use JDK 25 LTS (September 2025) — Spring Boot 4 minimum is Java 17, but we go with LTS-25 for long-term support.

Dockerfile для Spring Boot приложения

Multi-stage Dockerfile (оптимизированный для production)

Dockerfile for Spring Boot Application

Multi-stage Dockerfile (production-optimized)


# Stage 1: Builder (gradle контейнер)
FROM gradle:9.4.1-jdk25 AS builder

WORKDIR /app

# Копируем files для сборки
COPY . .

# Собираем JAR
RUN gradle bootJar -x test --no-daemon

# Stage 2: Runtime (минимальный JRE контейнер)
FROM eclipse-temurin:25-jre-noble

WORKDIR /app

# Копируем готовый JAR из builder stage
COPY --from=builder /app/build/libs/life-goals-api-*.jar app.jar

# Expose порт (информационно)
EXPOSE 8080

# Healthcheck
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD wget --quiet --tries=1 --spider http://localhost:8080/actuator/health || exit 1

# Запуск приложения
ENTRYPOINT ["java", "-jar", "app.jar"]

Docker Compose: Полное окружение

Полный docker-compose.yml для Life Goals Dashboard

Docker Compose: Complete Environment

Complete docker-compose.yml for Life Goals Dashboard


version: '3.8'

services:
  # PostgreSQL Database
  postgres:
    image: postgres:18-alpine
    container_name: lifegoals-postgres
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: lifegoals
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - lifegoals-network

  # Redis Cache
  redis:
    image: redis:8.6-alpine
    container_name: lifegoals-redis
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - lifegoals-network

  # Spring Boot API
  api:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: lifegoals-api
    ports:
      - "8080:8080"
    environment:
      DB_URL: jdbc:postgresql://postgres:5432/lifegoals
      DB_USERNAME: postgres
      DB_PASSWORD: postgres
      REDIS_URL: redis://redis:6379
      JWT_SECRET: your-secret-key
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    networks:
      - lifegoals-network

volumes:
  postgres_data:
  redis_data:

networks:
  lifegoals-network:
    driver: bridge

Volumes и Networks

Volumes: Persistent Data

Volume — это место на хосте, где контейнер может хранить данные. Данные в volume сохраняются, даже если контейнер удалён.

Volumes and Networks

Volumes: Persistent Data

Volume is a location on the host where a container can store data. Data in a volume persists even if the container is deleted.

Тип Синтаксис Использование Type Syntax Usage
Named Volume postgres_data:/var/lib/postgresql/data Database, cache. Docker управляет путём Named Volume postgres_data:/var/lib/postgresql/data Database, cache. Docker manages path
Bind Mount ./src:/app/src Разработка. Папка хоста монтируется в контейнер Bind Mount ./src:/app/src Development. Host folder mounts into container

Networks: Связь между контейнерами

В Compose, контейнеры могут общаться через имена сервисов (service names). Он автоматически создаёт bridge network.

Networks: Communication Between Containers

In Compose, containers can communicate via service names. It automatically creates a bridge network.

Healthcheck: Проверка здоровья

Healthcheck: Health Verification


# PostgreSQL healthcheck
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U postgres"]
  interval: 10s         # Проверять каждые 10 секунд
  timeout: 5s           # Ждать 5 секунд на ответ
  retries: 5            # Считать unhealthy после 5 неудачных
  start_period: 10s     # Не проверять первые 10 секунд

# Redis healthcheck
healthcheck:
  test: ["CMD", "redis-cli", "ping"]
  interval: 10s
  timeout: 5s
  retries: 5

Backlog #3: Docker Compose для локальной разработки 3 SP

Описание: Создайте docker-compose.yml с PostgreSQL 16, Redis 7, и локальной версией backend-приложения с mount точкой для горячей перезагрузки при изменении кода.

TDD тест (bash скрипт):

#!/bin/bash
# tests/docker-compose.sh

set -e

echo "Testing docker-compose setup..."

# Проверка, что docker-compose.yml существует
if [ ! -f "docker-compose.yml" ]; then
    echo "ERROR: docker-compose.yml not found"
    exit 1
fi

# Проверка структуры сервисов
echo "Checking docker-compose services..."

HAS_POSTGRES=$(grep -c "postgres:18" docker-compose.yml || true)
HAS_REDIS=$(grep -c "redis:8.6" docker-compose.yml || true)
HAS_BACKEND=$(grep -c "gradle bootRun" docker-compose.yml || true)

if [ "$HAS_POSTGRES" -eq 0 ]; then
    echo "ERROR: PostgreSQL 16 service not found"
    exit 1
fi

if [ "$HAS_REDIS" -eq 0 ]; then
    echo "ERROR: Redis 7 service not found"
    exit 1
fi

if [ "$HAS_BACKEND" -eq 0 ]; then
    echo "ERROR: Backend gradle bootRun service not found"
    exit 1
fi

# Проверка наличия mount volume для кода
echo "Checking for code mount volume..."
HAS_MOUNT=$(grep -c "\\.:" docker-compose.yml || true)
if [ "$HAS_MOUNT" -eq 0 ]; then
    echo "ERROR: Bind mount for source code not configured"
    exit 1
fi

# Стартуем сервисы в фоне
echo "Starting docker-compose services..."
docker-compose up -d

# Даём время на инициализацию
sleep 5

# Проверка, что сервисы running
echo "Verifying services are running..."
POSTGRES_UP=$(docker-compose ps | grep -c "postgres.*Up" || true)
REDIS_UP=$(docker-compose ps | grep -c "redis.*Up" || true)

if [ "$POSTGRES_UP" -eq 0 ]; then
    echo "ERROR: PostgreSQL container not running"
    docker-compose down
    exit 1
fi

if [ "$REDIS_UP" -eq 0 ]; then
    echo "ERROR: Redis container not running"
    docker-compose down
    exit 1
fi

# Проверка что PostgreSQL отвечает
echo "Testing PostgreSQL connection..."
PSQL_CHECK=$(docker-compose exec -T postgres \
    psql -U postgres -d lifegoals_dev -c "SELECT 1" 2>&1 || true)

if ! echo "$PSQL_CHECK" | grep -q "1"; then
    echo "ERROR: PostgreSQL not responding"
    docker-compose down
    exit 1
fi

# Проверка что Redis отвечает
echo "Testing Redis connection..."
REDIS_CHECK=$(docker-compose exec -T redis \
    redis-cli ping 2>&1 || true)

if [ "$REDIS_CHECK" != "PONG" ]; then
    echo "ERROR: Redis not responding"
    docker-compose down
    exit 1
fi

echo "SUCCESS: All docker-compose services verified and running"

# Cleanup
docker-compose down
exit 0
