Введение: PostgreSQL для Life Goals Dashboard

Introduction: PostgreSQL for Life Goals Dashboard

PostgreSQL is a powerful, reliable DBMS used in the Life Goals Dashboard backend. The project uses Spring Data JPA + Hibernate for ORM and Flyway for schema versioning.

Over 3 days you will learn SQL, JSONB for widget configuration, cursor-based pagination, Flyway migrations and optimization with indexes. All of this is necessary for working with data in LGD.

SQL for Life Goals Dashboard (Day 1)

Tables and Primary Keys

All tables in LGD use UUID for primary keys and have user_id and created_at fields. 


-- Таблица пользователей / User table
CREATE TABLE "user" (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- Таблица пищевых продуктов / Food products table
CREATE TABLE food_product (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    calories_per_100g INT NOT NULL,
    protein_per_100g DECIMAL(5, 2),
    carbs_per_100g DECIMAL(5, 2),
    fat_per_100g DECIMAL(5, 2),
    is_custom BOOLEAN DEFAULT false,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- Таблица виджетов пользователя / User dashboard widgets
CREATE TABLE user_dashboard_widget (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
    widget_type VARCHAR(50) NOT NULL,  -- INTERVIEW_PREP, FITNESS, BUDGET
    position INT NOT NULL,
    config JSONB NOT NULL,  -- Пользовательская конфигурация / User configuration
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(user_id, position)
);

SELECT, WHERE, JOIN, GROUP BY

SELECT, WHERE, JOIN, GROUP BY


-- Простой SELECT / Simple SELECT
SELECT id, name, calories_per_100g
FROM food_product
WHERE user_id = '123e4567-e89b-12d3-a456-426614174000';

-- WHERE с условиями / WHERE with conditions
SELECT *
FROM food_product
WHERE user_id = '123e4567-e89b-12d3-a456-426614174000'
  AND calories_per_100g > 100
  AND is_custom = false
ORDER BY calories_per_100g DESC;

-- JOIN между таблицами / JOIN between tables
SELECT f.name, f.calories_per_100g, u.email
FROM food_product f
JOIN "user" u ON f.user_id = u.id
WHERE u.email = 'alice@example.com';

-- GROUP BY для агрегации / GROUP BY for aggregation
SELECT
    user_id,
    COUNT(*) as total_products,
    AVG(calories_per_100g) as avg_calories
FROM food_product
WHERE is_custom = false
GROUP BY user_id
HAVING COUNT(*) > 5
ORDER BY avg_calories DESC;

-- LEFT JOIN для опциональных связей / LEFT JOIN for optional relationships
SELECT
    u.email,
    COUNT(fp.id) as product_count
FROM "user" u
LEFT JOIN food_product fp ON u.id = fp.user_id
GROUP BY u.id
ORDER BY product_count DESC;

Cursor-Based Pagination (День 2)

Почему не OFFSET?

OFFSET медленен на больших наборах данных. Проблема: при добавлении/удалении строк между запросами, позиции сдвигаются. Relay Connection pattern используют cursor-based pagination.

Cursor-Based Pagination (Day 2)

Why Not OFFSET?

OFFSET is slow on large datasets. Problem: when rows are added/deleted between requests, positions shift. Relay Connection pattern uses cursor-based pagination. 


-- Плохо (OFFSET) / Bad (OFFSET)
SELECT * FROM food_product
WHERE user_id = '123e4567-e89b-12d3-a456-426614174000'
ORDER BY created_at DESC
LIMIT 20 OFFSET 40;  -- Must read first 40, then next 20

-- Хорошо (Keyset/Cursor-based) / Good (Keyset/Cursor-based)
SELECT * FROM food_product
WHERE user_id = '123e4567-e89b-12d3-a456-426614174000'
  AND (created_at, id) < ('2025-01-15 10:30:00', 'last-id-from-previous-page')
ORDER BY created_at DESC, id DESC
LIMIT 21;  -- 20 + 1 to determine hasNextPage

-- Cursor = BASE64(created_at || ',' || id)
-- Decode on client and use in next request

Реализация Relay Connection

Relay Connection Implementation


-- Таблица для примера / Example table
CREATE TABLE workout (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES "user"(id),
    exercise VARCHAR(255) NOT NULL,
    duration_minutes INT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

-- Запрос для первой страницы / Query for first page
SELECT
    id,
    exercise,
    duration_minutes,
    created_at,
    -- Генерируем cursor / Generate cursor
    encode(
        (created_at::TEXT || ',' || id::TEXT)::BYTEA,
        'base64'
    ) as cursor
FROM workout
WHERE user_id = '123e4567-e89b-12d3-a456-426614174000'
ORDER BY created_at DESC, id DESC
LIMIT 21;  -- 20 + 1

-- Запрос со следующей страницы / Query for next page
SELECT
    id,
    exercise,
    duration_minutes,
    created_at,
    encode(
        (created_at::TEXT || ',' || id::TEXT)::BYTEA,
        'base64'
    ) as cursor
FROM workout
WHERE user_id = '123e4567-e89b-12d3-a456-426614174000'
  AND (created_at, id) < ('2025-01-15 10:30:00'::TIMESTAMP, 'abc-123'::UUID)
ORDER BY created_at DESC, id DESC
LIMIT 21;
Почему это работает лучше: Cursor-based пагинация не зависит от количества пропущенных строк. Даже если у пользователя 1 миллион тренировок, запрос выполняется за одно время.
Why this works better: Cursor-based pagination doesn't depend on the number of skipped rows. Even if a user has 1 million workouts, the query executes in the same time.

Flyway: Версионирование схемы (День 2-3)

Что такое Flyway?

Flyway — это инструмент миграции БД, который версионирует схему. Каждая миграция — это файл SQL с именем вида V{N}__{description}.sql.

Создание миграций

Flyway: Schema Versioning (Day 2-3)

What is Flyway?

Flyway is a database migration tool that versions the schema. Each migration is a SQL file named V{N}__{description}.sql.

Creating Migrations


-- Файл / File: src/main/resources/db/migration/V1__initial_schema.sql
-- Первая миграция / First migration

CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

CREATE TABLE "user" (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE food_product (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    name VARCHAR(255) NOT NULL,

-- Файл / File: src/main/resources/db/migration/V2__add_food_nutrition.sql
ALTER TABLE food_product
ADD COLUMN protein_per_100g DECIMAL(5, 2),
ADD COLUMN carbs_per_100g DECIMAL(5, 2),

-- Файл / File: src/main/resources/db/migration/V3__add_widget_table.sql

Лучшие практики индексирования

Indexing Best Practices

  • Индексируйте foreign keys: `user_id` часто используется в WHERE
  • Индексируйте уникальные поля: `email` для быстрого поиска пользователя
  • Composite индексы: `(user_id, created_at)` для cursor-based pagination
  • Избегайте лишних индексов: Каждый индекс замедляет INSERT/UPDATE/DELETE
  • JSONB индексы: Используйте GIN для часто ищущихся полей в config
  • Index foreign keys: `user_id` is often used in WHERE
  • Index unique fields: `email` for fast user lookup
  • Composite indexes: `(user_id, created_at)` for cursor-based pagination
  • Avoid unnecessary indexes: Each index slows INSERT/UPDATE/DELETE
  • JSONB indexes: Use GIN for frequently searched fields in config

Что нового в PostgreSQL 18 / What's New in PostgreSQL 18

Life Goals Dashboard использует PostgreSQL 18.3 — последнюю версию с критическими улучшениями производительности и функциональности.

Async I/O Subsystem (новый в PG 18)

PostgreSQL 18 вводит встроенную асинхронную I/O подсистему с поддержкой io_method: worker и io_uring (Linux). Это может улучшить производительность до 3× быстрее на NVMe дисках за счёт параллельного чтения блоков вместо синхронного I/O. 


# Configure async I/O in postgresql.conf
backend_io_method = worker     # or io_uring on Linux 6.0+
backend_io_worker_threads = 4  # Adjust based on CPU cores

Встроенная функция uuidv7() — Время-упорядоченные UUID

В PostgreSQL 18 добавлена встроенная функция uuidv7(), которая генерирует время-упорядоченные UUID (RFC 4122 v7). В отличие от gen_random_uuid(), которые случайны, uuidv7() гарантирует монотонный порядок по времени вставки. Это существенно улучшает производительность B-tree индексов для первичных ключей.

Ключевое отличие: B-tree вставки со случайными UUID требуют перебалансировки, но с uuidv7() новые элементы добавляются всегда в конец листа. 


-- Старый подход (случайный UUID, плохо для B-tree):
CREATE TABLE food_product (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ...
);

-- Новый подход (время-упорядоченный UUID, отличная локальность):
CREATE TABLE food_product (
    id UUID PRIMARY KEY DEFAULT uuidv7(),
    ...
);

-- uuidv7() гарантирует, что новые строки вставляются
-- в растущий конец B-tree листа (sequential I/O)

Virtual Generated Columns (GENERATED ALWAYS AS ... VIRTUAL)

PostgreSQL 18 вводит virtual (not materialized) generated columns. Это автоматически вычисляемые столбцы, которые не занимают место на диске — значение вычисляется при каждом чтении. 


-- Пример: вычисляемая общая калорийность вещества
CREATE TABLE meal_entry (
    id UUID PRIMARY KEY DEFAULT uuidv7(),
    user_id UUID NOT NULL,
    product_id UUID,
    calories_per_100g NUMERIC NOT NULL,
    portion_g NUMERIC NOT NULL,

    -- Virtual column: вычисляется при чтении, не хранится
    total_calories NUMERIC GENERATED ALWAYS AS
        (calories_per_100g * portion_g / 100)
    VIRTUAL,

    created_at TIMESTAMP DEFAULT NOW()
);

-- При запросе total_calories автоматически пересчитывается:
SELECT id, portion_g, calories_per_100g, total_calories
FROM meal_entry WHERE user_id = $1;
-- total_calories = calories_per_100g * portion_g / 100 (on-the-fly)

Skip Scan на Multi-Column B-tree Индексах

PostgreSQL 18 добавляет оптимизацию skip scan для составных индексов. Если индекс (user_id, created_at) и запрос ищет created_at = X без указания user_id, планировщик может "пропустить" первый столбец и перейти напрямую ко второму. 


-- Составной индекс
CREATE INDEX idx_user_created ON meal_entry(user_id, created_at);

-- Запрос без user_id теперь может использовать skip scan:
SELECT * FROM meal_entry WHERE created_at > '2026-01-01';
-- PostgreSQL 18 может использовать индекс вместо seq scan

OAuth 2.0 Authentication in Core

PostgreSQL 18 добавляет встроенную поддержку OAuth 2.0 для аутентификации пользователей, снимая необходимость в отдельных расширениях.

Improved Parallel Queries и Partitioning

Параллельные запросы и динамическое партиционирование стали ещё более эффективны. Для больших наборов данных (например, истории приёма пищи за многие месяцы) это даёт значительный прирост производительности.

What's New in PostgreSQL 18

Life Goals Dashboard uses PostgreSQL 18.3 — the latest version with critical performance and feature improvements.

Async I/O Subsystem (new in PG 18)

PostgreSQL 18 introduces built-in asynchronous I/O support with io_method: worker and io_uring (Linux). This can improve performance up to 3× faster on NVMe drives via parallel block reads instead of synchronous I/O. 


# Configure async I/O in postgresql.conf
backend_io_method = worker     # or io_uring on Linux 6.0+
backend_io_worker_threads = 4  # Adjust based on CPU cores

Built-in uuidv7() Function — Time-Ordered UUIDs

PostgreSQL 18 adds the uuidv7() function, generating time-ordered UUIDs (RFC 4122 v7). Unlike gen_random_uuid(), which are random, uuidv7() guarantees monotonic order by insertion time. This dramatically improves B-tree index performance for primary keys.

Key difference: B-tree inserts with random UUIDs require rebalancing, but with uuidv7() new entries always append to the leaf end. 


-- Old approach (random UUID, bad for B-tree):
CREATE TABLE food_product (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ...
);

-- New approach (time-ordered UUID, excellent locality):
CREATE TABLE food_product (
    id UUID PRIMARY KEY DEFAULT uuidv7(),
    ...
);

-- uuidv7() guarantees new rows insert
-- at the growing B-tree leaf end (sequential I/O)

Virtual Generated Columns (GENERATED ALWAYS AS ... VIRTUAL)

PostgreSQL 18 introduces virtual (not materialized) generated columns. These are automatically computed columns that take no storage — the value is computed at read time. 


-- Example: computed total calorie count
CREATE TABLE meal_entry (
    id UUID PRIMARY KEY DEFAULT uuidv7(),
    user_id UUID NOT NULL,
    product_id UUID,
    calories_per_100g NUMERIC NOT NULL,
    portion_g NUMERIC NOT NULL,

    -- Virtual column: computed at read time, not stored
    total_calories NUMERIC GENERATED ALWAYS AS
        (calories_per_100g * portion_g / 100)
    VIRTUAL,

    created_at TIMESTAMP DEFAULT NOW()
);

-- Querying total_calories auto-recomputes:
SELECT id, portion_g, calories_per_100g, total_calories
FROM meal_entry WHERE user_id = $1;
-- total_calories = calories_per_100g * portion_g / 100 (on-the-fly)

Skip Scan on Multi-Column B-tree Indexes

PostgreSQL 18 adds skip scan optimization for composite indexes. If an index is (user_id, created_at) and a query searches created_at = X without specifying user_id, the planner can "skip" the first column and jump directly to the second. 


-- Composite index
CREATE INDEX idx_user_created ON meal_entry(user_id, created_at);

-- Query without user_id can now use skip scan:
SELECT * FROM meal_entry WHERE created_at > '2026-01-01';
-- PostgreSQL 18 may use index instead of seq scan

OAuth 2.0 Authentication in Core

PostgreSQL 18 adds built-in OAuth 2.0 support for user authentication, removing the need for separate extensions.

Improved Parallel Queries and Partitioning

Parallel queries and dynamic partitioning are now even more efficient. For large datasets (e.g., meal history over many months), this provides significant performance gains.
