Введение: Git для Life Goals DashboardIntroduction: Git for Life Goals Dashboard

За 2 дня вы освоите основные концепции Git, практикуете реальные сценарии разработки и научитесь следовать стандартам проекта. Это критически важно для слияния вашего кода в `main` веку.

Over 2 days, you'll master Git fundamentals, practice real development scenarios, and learn to follow project standards. This is critical for merging your code into the `main` branch.

Git Базовые концепцииGit Basics

Три основных состояния файловThree File States

Git отслеживает файлы в трёх местах:

  • Working Directory — папка на вашем компьютере. Файлы здесь могут быть изменены, но Git их ещё не отслеживает.
  • Staging Area (Index) — промежуточная зона. Вы добавляете сюда изменения перед коммитом через `git add`.
  • Repository (.git) — база данных истории. После `git commit` изменения сохраняются здесь навсегда.

Поток выглядит так:

Git tracks files in three locations:

  • Working Directory — folder on your computer. Files here can be modified but Git doesn't track them yet.
  • Staging Area (Index) — intermediate zone. You add changes here before committing via `git add`.
  • Repository (.git) — history database. After `git commit`, changes are saved here permanently.

The flow looks like this:


# 1. Вы редактируете файлы в Working Directory
$ nano src/index.ts

# 2. Проверьте статус
$ git status
# Output: modified: src/index.ts (не добавлено в staging)

# 3. Добавьте в Staging Area
$ git add src/index.ts

# 4. Проверьте staging
$ git status
# Output: new file: src/index.ts (готово к коммиту)

# 5. Создайте коммит
$ git commit -m "feat(dashboard): add widget registry"

# 6. Коммит сохранён в Repository. История обновлена.

Ветвление и слияниеBranching & Merging

Что такое ветка (branch)?What is a Branch?

Ветка — это независимый "поток" разработки. Главная ветка `main` содержит стабильный, готовый к продакшену код. Для каждой новой функции вы создаёте отдельную ветку, работаете независимо, а затем сливаете обратно в `main`.

A branch is an independent "stream" of development. The main branch `main` contains stable, production-ready code. For each new feature, you create a separate branch, work independently, then merge it back to `main`. 


# Просмотр локальных веток
git branch

# Просмотр удалённых веток
git branch -r

# Просмотр всех веток
git branch -a

# Создание новой ветки
git branch feature/LGD-123-add-notifications

# Переключение на ветку
git checkout feature/LGD-123-add-notifications

# Создание и переключение в одной команде (рекомендуется)
git checkout -b feature/LGD-123-add-notifications

# Удаление локальной ветки
git branch -d feature/LGD-123-add-notifications

# Удаление удалённой ветки
git push origin --delete feature/LGD-123-add-notifications

Merge vs RebaseMerge vs Rebase

Merge — объединение двух веток с созданием "merge commit". История остаётся полной, но граф может быть запутанным.

Merge — combining two branches by creating a "merge commit". History remains full but the graph can be messy. 


# Вы на ветке main
git pull origin main              # Обновляем main

# Переключаемся на свою ветку
git checkout feature/my-feature

# Мержим main в свою ветку (решаем конфликты локально)
git merge main

# Или наоборот: переключаемся на main и мержим туда вашу ветку
git checkout main
git merge feature/my-feature

Rebase — "перемещение" вашей ветки на новый основание. История остаётся линейной и чистой. Лучше для локальных веток, до создания PR.

Rebase — "moving" your branch to a new base. History remains linear and clean. Better for local branches, before creating a PR. 


# Вы на ветке feature/my-feature
git fetch origin                  # Получаем последние изменения с сервера
git rebase origin/main            # Перемещаем свои коммиты на top of main

# Если есть конфликты, Git остановит процесс
# Вы разрешите конфликты в файлах, затем
git add resolved-file.ts
git rebase --continue

# Если хотите отменить rebase
git rebase --abort
Best Practice для LGD: Перед созданием PR используйте `git rebase origin/main` на вашей ветке. Это обеспечивает чистую, линейную историю и облегчает code review.
Best Practice for LGD: Before creating a PR, use `git rebase origin/main` on your branch. This ensures clean, linear history and makes code review easier.

Разрешение конфликтовResolving Conflicts

Конфликт возникает, когда одна строка изменена в обеих ветках. Git не может автоматически решить, какую версию оставить.

A conflict occurs when the same line is modified in both branches. Git cannot automatically decide which version to keep. 


// Пример конфликта в файле
<<<<<<< HEAD
const targetCurrency = "EUR";  // ваше изменение (main)
=======
const targetCurrency = "USD";  // изменение из другой ветки
>>>>>>> feature/budget-currencies

Вы должны вручную выбрать, какое изменение оставить (или скомбинировать оба):

You must manually choose which change to keep (or combine both): 


// После разрешения конфликта (удалите маркеры)
const targetCurrency = "EUR";  // вы выбрали это

Затем сообщите Git, что конфликт разрешён:

Then tell Git the conflict is resolved: 


git add conflicted-file.ts
git commit -m "merge: resolve currency conflict"
# или, если это rebase:
git rebase --continue

Conventional CommitsConventional Commits

Каждый коммит должен следовать формату Conventional Commits. Это позволяет автоматизировать изменение версии (версионирование семантики) и генерировать changelog.

Формат:

Each commit must follow the Conventional Commits format. This enables automated semantic versioning and changelog generation.

Format:


type(scope): subject

body (optional)

footer (optional)

Типы:

Types:

  • feat — новая функцияnew feature
  • fix — исправление ошибкиbug fix
  • refactor — переработка кодаcode refactoring
  • test — добавление тестовadding tests
  • docs — обновление документацииdocumentation update
  • chore — обновление зависимостей, конфига и т.д.dependency update, config changes, etc.
  • perf — улучшение производительностиperformance improvement

Примеры:

Examples:


# Просто функция
git commit -m "feat(fitness): add macro ratio calculator"

# С дополнительной информацией
git commit -m "fix(budget): correct KRW/EUR exchange rate cache

Previously, the exchange rate cache key did not include the currency pair,
causing all pairs to share the same cached value.

Fixes #45"

# Breaking change
git commit -m "refactor(common)!: redesign WidgetProvider interface

BREAKING CHANGE: validateConfig() now returns ValidationError[] instead of Map"

Продвинутые техникиAdvanced Techniques

Git Stash — сохранение незавершённой работыGit Stash — Saving Work in Progress

Если вы случайно начали работу не в той ветке, или нужно срочно переключиться на другую задачу, используйте git stash для сохранения изменений во временное хранилище.

If you accidentally started work on the wrong branch, or need to urgently switch to another task, use git stash to save changes to temporary storage. 


# Вы на ветке feature/A, начали делать изменения
git status
# On branch feature/A
# modified: file.ts

# Срочно нужно переключиться на feature/B
# Сначала сохраните изменения в stash
git stash

# Или с описанием
git stash save "WIP: meal nutrition calculator"

# Переключитесь на другую ветку
git checkout feature/LGD-20-urgent-fix

# Когда вернулись на feature/A
git checkout feature/A

# Восстановите stash
git stash pop

# Или просто просмотрите список stash'ей
git stash list
# Output:
# stash@{0}: WIP: meal nutrition calculator
# stash@{1}: On feature/A: another change

# Восстановите конкретный stash
git stash apply stash@{0}

# Удалите stash
git stash drop stash@{0}

Git Reset — отмена коммитовGit Reset — Undoing Commits

git reset имеет три режима, влияющие на то, что происходит с вашими изменениями:

git reset has three modes affecting what happens with your changes: 


# Soft reset: отмена коммита, но изменения остаются в staging
git reset --soft HEAD~1

# Mixed reset (по умолчанию): отмена коммита, изменения в working dir
git reset --mixed HEAD~1
git reset HEAD~1  # То же самое

# Hard reset: отмена коммита И всех изменений (опасно!)
git reset --hard HEAD~1

# Откат конкретного файла к версии из последнего коммита
git checkout HEAD -- file.ts

# Отмена последнего commit, если ещё не pushed
git reset --hard HEAD~1

# Если уже pushed, используйте revert (создаёт новый commit)
git revert HEAD
Осторожно с --hard: `git reset --hard` уничтожает изменения навсегда. Убедитесь, что вы хотите это сделать. Если вы случайно удалили важный код, попробуйте `git reflog` найти старый HEAD.
Caution with --hard: `git reset --hard` destroys changes permanently. Make sure you want to do this. If you accidentally deleted important code, try `git reflog` to find the old HEAD.

Git Bisect — поиск коммита, который внёс багGit Bisect — Finding the Commit That Introduced a Bug

git bisect использует бинарный поиск, чтобы найти точный коммит, который сломал функцию.

git bisect uses binary search to find the exact commit that broke a feature. 


# Начните bisect
git bisect start

# Укажите "плохой" коммит (текущий)
git bisect bad

# Укажите последний "хороший" коммит
git bisect good v1.0

# Git проверит коммит в середине. Протестируйте.
# Если баг есть:
git bisect bad

# Если баг нет:
git bisect good

# Повторяйте до поиска конкретного коммита
# После того как найден:
git bisect reset

Git Cherry-pick — применение конкретного коммитаGit Cherry-pick — Applying a Specific Commit

Иногда нужно применить одиночный коммит из другой ветки в вашу текущую ветку.

Sometimes you need to apply a single commit from another branch to your current branch. 


# Вы на ветке feature/my-branch
# Вам нужен коммит abc123 из main

# Скопируйте коммит в текущую ветку
git cherry-pick abc123

# Если конфликты:
# 1. Разрешите конфликты
# 2. Добавьте файлы
git add resolved-file.ts

# 3. Закончите cherry-pick
git cherry-pick --continue

# Или отмените
git cherry-pick --abort

Тесты (запустить этот bash скрипт, чтобы проверить .gitignore):

Create a correct `.gitignore` file for the Life Goals Dashboard project that includes rules for: Kotlin/Gradle builds, Node.js frontend, IDE configurations, and excludes files that should be committed (e.g., `__generated__/` and `.env.example`).

Tests (run this bash script to check .gitignore):

#!/bin/bash
# test_gitignore.sh — Проверка корректности .gitignore

ERRORS=0

# Test 1: .gitignore exists
if [ ! -f ".gitignore" ]; then
    echo "FAIL: .gitignore file not found"
    ERRORS=$((ERRORS + 1))
else
    echo "PASS: .gitignore exists"
fi

# Test 2: Kotlin/Gradle ignores
for pattern in "build/" ".gradle/" "*.jar" "*.class"; do
    if ! grep -q "$pattern" .gitignore 2>/dev/null; then
        echo "FAIL: Missing pattern: $pattern"
        ERRORS=$((ERRORS + 1))
    else
        echo "PASS: Contains $pattern"
    fi
done

# Test 3: Node/Frontend ignores
for pattern in "node_modules/" "dist/" ".env"; do
    if ! grep -q "$pattern" .gitignore 2>/dev/null; then
        echo "FAIL: Missing pattern: $pattern"
        ERRORS=$((ERRORS + 1))
    else
        echo "PASS: Contains $pattern"
    fi
done

# Test 4: IDE ignores
for pattern in ".idea/" ".vscode/" "*.iml"; do
    if ! grep -q "$pattern" .gitignore 2>/dev/null; then
        echo "FAIL: Missing IDE pattern: $pattern"
        ERRORS=$((ERRORS + 1))
    else
        echo "PASS: Contains $pattern"
    fi
done

# Test 5: __generated__/ should NOT be ignored (Relay output is committed)
if grep -q "^__generated__" .gitignore 2>/dev/null; then
    echo "FAIL: __generated__/ should NOT be in .gitignore (Relay output is committed)"
    ERRORS=$((ERRORS + 1))
else
    echo "PASS: __generated__/ not ignored (correct)"
fi

# Test 6: .env.example should NOT be ignored
if grep -q "^\.env\.example" .gitignore 2>/dev/null; then
    echo "FAIL: .env.example should NOT be in .gitignore"
    ERRORS=$((ERRORS + 1))
else
    echo "PASS: .env.example not ignored (correct)"
fi

if [ $ERRORS -eq 0 ]; then
    echo -e "\n✅ ALL TESTS PASSED"
else
    echo -e "\n❌ $ERRORS TEST(S) FAILED"
    exit 1
fi
